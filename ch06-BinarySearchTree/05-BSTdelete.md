## 6. 삭제 연산 (Deletion)
이진 탐색 트리에서 노드를 삭제할 때는 삭제 후에도 이진 탐색 트리의 조건( $Key(L) < Key(n) < Key(R)$ ) 이 유지되도록  
부모와 자식 노드들을 재연결하는 것이 핵심입니다.  

### 6-1. 삭제를 위한 보조 연산
임의의 노드를 삭제하기 위해 선행되어야 하거나 내부적으로 사용되는 연산들입니다.  

#### ① 최솟값 찾기 (min)
* **원리**: 루트(또는 특정 서브트리)에서 시작하여 **왼쪽 자식**만을 계속 따라 내려가다가 `null`을 만나기 직전의 노드를 반환합니다.  
* **용도**: 삭제 연산(특히 Case 3)에서 후계자 노드를 찾을 때 사용됩니다.  

#### ② 최솟값 삭제 (deleteMin)  
* **원리**: 최솟값 노드 $x$를 찾은 후, $x$의 부모 노드 $p$와 $x$의 **오른쪽 자식 노드**를 직접 연결합니다.  
* **특징**: 최솟값 노드는 왼쪽 자식이 없으므로, 오른쪽 자식의 존재 여부와 상관없이 연결하면 논리적으로 완결됩니다.

### 6-2. 삭제 연산의 3가지 케이스 
삭제하고자 하는 노드 $r$의 자식 노드 개수에 따라 처리 방식이 달라집니다.    

| 케이스 | 상황 | 처리 방법 |  
| :--- | :--- | :--- |  
| **Case 1** | **자식이 없는 경우** (Leaf) | 부모 노드에서 $r$을 가리키던 레퍼런스를 `null`로 변경 |  
| **Case 2** | **자식이 하나인 경우** | $r$의 부모가 $r$의 유일한 자식을 직접 연결 (r은 버려짐) |  
| **Case 3** | **자식이 둘인 경우** | $r$의 자리에 **후계자 노드** $m$을 이동시키고 후계자 $m$이 있던 자리에 후계자 $m$의 자식을 연결 |  

<img width="854" height="362" alt="image" src="https://github.com/user-attachments/assets/f5186bb4-f1eb-4eed-9bba-5c0c9ed99acc" />  


### 6-3. Case 3 (자식이 둘인 경우) 분석
부모 노드는 하나인데 연결해야 할 자식은 둘이므로 단순히 연결할 수 없습니다.    
따라서 트리의 정렬 상태를 유지할 수 있는 대체 노드(후계자)를 찾아야 합니다.    

> **후계자 선택 방법**  
> 1. **중위 후속자 (Inorder Successor)**: 오른쪽 서브트리에서 가장 작은 값 (오른쪽 서브트리의 `min`)   
> 2. **중위 선행자 (Inorder Predecessor)**: 왼쪽 서브트리에서 가장 큰 값 (왼쪽 서브트리의 `max`)   
>  *보통은 오른쪽 서브트리의 최솟값(Successor)을 사용하는 방식을 많이 택합니다.*   

#### Case 3 처리 단계 (오른쪽 최솟값 선택 시):   
1. 삭제할 노드 $r$의 오른쪽 서브트리에서 최솟값 노드 $m$을 찾습니다.   
2. $s$의 값을 $r$의 자리로 복사(또는 노드 교체)합니다.   
3. 원래 $s$가 있던 자리는 **Case 1** 또는 **Case 2**의 방식으로 삭제합니다 (최솟값 노드는 자식이 최대 하나뿐이므로 처리가 쉽습니다).

### 6-4. Java 구현 로직 분석  
```java
public Node delete(Node n, Key k) {
    if (n == null) return null;
    int t = n.getKey().compareTo(k);
    if (t > 0) n.setLeft(delete(n.getLeft(), k));
    else if (t < 0) n.setRight(delete(n.getRight(), k));
    else { // 삭제할 노드 발견
        if (n.getRight() == null) return n.getLeft(); // Case 0, 1, 2-1
        if (n.getLeft() == null) return n.getRight(); // Case 2-2
        
        // Case 3
        Node target = n;
        n = min(target.getRight()); // 후계자 찾기
        n.setRight(deleteMin(target.getRight())); // 후계자 자리 정리 후 연결
        n.setLeft(target.getLeft());
    }
    return n;
}
```

재귀 호출을 이용한 `delete(n, k)`의 내부 흐름입니다.  

1.  **위치 탐색**: $k$가 현재 키보다 작으면 왼쪽, 크면 오른쪽 서브트리로 이동합니다.  
2.  **노드 발견**: 삭제할 노드에 도착하면 자식 상태를 체크합니다.  
    * `right == null`이면 `left` 반환 (Case 0, 1, 2-1 처리)  
    * `left == null`이면 `right` 반환 (Case 2-2 처리)  
    * **둘 다 있으면**:   
        * `target`에 현재 노드 복사  
        * 현재 노드를 `min(target.right)`로 교체  
        * 새 노드의 오른쪽 자식을 `deleteMin(target.right)` 결과로 연결   
        * 새 노드의 왼쪽 자식을 `target.left`로 유지

### 6-5. 시간 복잡도 (Complexity)  
* **삭제 연산의 복잡도**: $O(h)$ (여기서 $h$는 트리의 높이)  
* **이유**: 삭제 대상을 찾는 탐색 과정( $O(h)$ ) + 후계자를 찾는 과정( $O(h)$ ) + 링크 재연결( $O(1)$ )의 합이므로 결국 트리의 높이에 비례합니다.  
* **성능**: 탐색, 삽입과 마찬가지로 균형 잡힌 트리에서는 $O(\log N)$ 을 보장하지만, 편향된 트리에서는 $O(N)$ 까지 성능이 저하됩니다.

## 7. 이진 탐색 트리 연산의 수행 시간 분석  
이진 탐색 트리에서 **탐색(Search), 삽입(Insert), 삭제(Delete)** 연산은 모두 트리의 경로를 따라 이동하므로, 공통적인 성능 특성을 공유합니다.  

### 7-1. 연산 비용의 구성 (Cost Breakdown)  
모든 연산은 루트에서 시작하여 특정 노드나 단말(Leaf) 노드까지 내려가는 과정을 포함합니다.  

* **하향 이동 (Descending Path)**: 루트에서 목표 지점까지 내려갈 때 각 층마다 재귀 호출이 발생합니다. 각 층에서의 비교 및 이동 작업은 $O(1)$입니다.  
* **상향 이동 (Ascending Path)**: 삽입과 삭제 연산의 경우, 작업 완료 후 다시 루트까지 거슬러 올라오며  
  `setLeft()`, `setRight()` 등의 참조 갱신을 수행합니다. 각 층당 $O(1)$의 시간이 소요됩니다.    
* **종합**: 전체 수행 시간은 방문하는 노드의 개수, 즉 트리의 높이($h$) 에 비례하는 $O(h)$가 됩니다.    

### 7-2. 트리의 높이( $h$ )에 대한 분석   
노드의 개수가 $N$개일 때, 트리의 높이 $h$는 트리의 모양(포화도)에 따라 결정됩니다.  

#### ① 높이의 범위 (Height Bound)  
이진 트리의 높이 $h$ 는 수학적으로 다음과 같은 범위를 가집니다.  
$$\lceil \log_2(N+1) \rceil \le h \le N$$    
  
* **최소 높이** ( $h \approx \log_2 N$ ): 트리가 **완전 이진 트리(Complete Binary Tree)** 형태를 띠어 모든 층이 빽빽하게 채워졌을 때입니다.  
* **최대 높이** ( $h = N$ ): 데이터가 정렬된 순서대로 삽입되어 한쪽으로만 길게 늘어진 **편향 이진 트리(Skewed Binary Tree)** 형태일 때입니다.

#### ② 평균 높이 (Average Case)
비어 있는 이진 탐색 트리에 $N$개의 키를 **랜덤한 순서** 로 삽입한다고 가정할 때, 트리의 평균 높이에 대한 수학적 기대치는 다음과 같습니다.
$$h \approx 1.39 \log_2 N$$  
이는 무작위로 데이터를 넣기만 해도 어느 정도 균형이 잡혀 $O(\log N)$ 에 가까운 효율적인 성능을 기대할 수 있음을 시사합니다.

### 7-3. 연산별 시간 복잡도 요약
트리의 높이를 $h$ , 노드 수를 $N$ 이라 할 때의 복잡도입니다.  
  
| 연산 | 평균 시간 복잡도 | 최악의 시간 복잡도 (편향 트리) | 비고 |   
| :--- | :--- | :--- | :--- |   
| **탐색 (Search)** | **$O(\log N)$** | **$O(N)$** | 루트에서 단말까지 하향 |   
| **삽입 (Insert)** | **$O(\log N)$** | **$O(N)$** | 하향 탐색 + 상향 재연결 |   
| **삭제 (Delete)** | **$O(\log N)$** | **$O(N)$** | 하향 탐색 + 후계자 탐색 + 상향 재연결 |   

### 7-4. 결론 및 보완점    
이진 탐색 트리는 매우 효율적인 자료구조이지만, **데이터가 삽입되는 순서에 따라 성능이 극단적으로 변한다**는 치명적인 약점이 있습니다.    

* **문제점**: 이미 정렬된 데이터(예: 1, 2, 3, 4, 5...)를 순서대로 넣으면 트리가 선형 리스트가 되어 검색 효율이 완전히 파괴됩니다.  
  이렇게 만들어진 Skewed Tree는 정렬 안 된 배열에서 맨 뒤까지 다 찾아보는 순차 탐색( $O(N)$ )이랑 같은 시간이 소요 됩니다. 
* **해결책**: 이를 방지하기 위해 삽입/삭제 시 스스로 트리의 높이를 제한하여  
  항상 $O(\log N)$을 보장하게 만드는 **자가 균형 이진 탐색 트리(Self-Balancing BST)** 인 **AVL 트리**나 **레드-블랙 트리** 를 사용하게 됩니다.
