## 2-1. 2-3-4 트리 (2-3-4 Tree)  
2-3-4 트리는 차수가 4인 B-트리의 특수한 형태로, 모든 내부 노드가 2, 3, 또는 4개의 자식을 가질 수 있는 트리입니다.  

* **2-노드**: 1개의 키, 2개의 자식  
* **3-노드**: 2개의 키, 3개의 자식  
* **4-노드**: 3개의 키, 4개의 자식  
* **특징**: 삽입 시 루트에서 리프 방향으로 내려가며 4-노드를 미리 분할(Preemptive Split)하여 효율성을 높입니다.  

## 2-2. B-트리의 정의
차수가 $M$인 B-트리는 다음의 **5가지 핵심 성질**을 만족하는 다방향 탐색 트리입니다.  
<img width="852" height="270" alt="image" src="https://github.com/user-attachments/assets/8e01e0da-12dc-45cb-8f7c-b9badb8ba089" />  


1.  **모든 노드의 자식 수**: 각 노드는 최대 $M$개의 자식을 가질 수 있다.  
2.  **최소 자식 수**: 루트 노드와 리프 노드를 제외한 모든 내부 노드는 최소 $\lceil M/2 \rceil$개의 자식을 가져야 한다.  
3.  **루트 노드의 조건**: 루트 노드가 리프가 아니라면 최소 2개의 자식을 가져야 한다.   
4.  **키의 개수**: $K$개의 키를 가진 노드는 항상 $K+1$개의 자식을 가진다.  
     (즉, 노드 내 키의 개수는 최소 $\lceil M/2 \rceil - 1$개, 최대 $M-1$개여야 한다.)    
5.  **완전 균형**: 모든 리프 노드는 항상 **동일한 레벨(깊이)** 에 존재해야 한다.    

## 2-3. '키(Key)'의 정의와 역할  

교과서적인 관점에서 **키(Key)** 는 단순한 숫자가 아니라 다음과 같은 의미를 갖습니다.  

* **정의**: 레코드를 식별할 수 있는 유일한 값을 가진 데이터 필드입니다.  
* **역할 (분기점)**: 노드 내에서 자식 노드로 내려가는 경로를 결정하는 **이정표** 역할을 합니다.   
    * 예: 키 $k_1, k_2$가 있는 노드에서 첫 번째 자식은 $k_1$보다 작은 값,  
      두 번째 자식은 $k_1$과 $k_2$ 사이의 값, 세 번째 자식은 $k_2$보다 큰 값을 저장합니다.  
* **물리적 구조**: 실제 구현에서는 `Key`와 함께 해당 데이터가 저장된 주소를 가리키는 `Pointer`가 한 쌍으로 저장됩니다.  

## 2-4. 차수($M$)에 따른 노드 제약 조건 수식  
B-트리의 핵심은 **"노드를 절반 이상 채워야 한다"** 는 규칙입니다. 이를 수학적으로 정리하면 다음과 같습니다.  

| 항목 | 계산 수식 | $M=3$ (2-3 트리) | $M=5$ (B-트리 예시) |  
| :--- | :--- | :---: | :---: |  
| **최대 자식 수** | $M$ | 3 | 5 |  
| **최대 키 수** | $M-1$ | 2 | 4 |  
| **최소 자식 수** | $\lceil M/2 \rceil$ | 2 | 3 |  
| **최소 키 수** | $\lceil M/2 \rceil - 1$ | 1 | 2 |  

> **왜 절반 이상(Half-full) 채워야 하나요?**  
> 노드가 너무 비어 있으면 트리의 높이가 높아져서 디스크 접근 횟수가 늘어납니다.  
> 반대로 너무 꽉 차 있으면 삽입 시 분리가 너무 자주 일어납니다. $\lceil M/2 \rceil$ 규칙은  
> **저장 공간의 효율성**과 **탐색 성능** 사이의 최적의 균형점을 수학적으로 강제하는 것입니다.  

## 2-5. B-트리의 탐색 연산 (Search Operation)  
B-트리에서의 탐색은 이진 탐색 트리의 확장된 형태이며, 대량의 데이터를 다루는 특성에 최적화되어 있습니다.
<img width="1161" height="382" alt="image" src="https://github.com/user-attachments/assets/46d7f9cd-79b2-4bdb-9add-30a2c9686c61" />  


* **루트 노드 시작**: 항상 루트 노드에서 탐색을 시작합니다.
* **노드 내 탐색**: 방문한 노드에 저장된 키들과 탐색하고자 하는 키를 비교합니다. 
* **이진 탐색(Binary Search)의 활용**: 
    - B-트리의 노드는 일반적으로 수백 개 이상의 키를 가질 수 있습니다. 
    - 따라서 노드 내부에서 선형 탐색($O(M)$)을 수행하는 대신 **이진 탐색($O(\log M)$)**을 수행하여 적절한 서브트리 경로를 빠르게 찾아냅니다.
* **경로 결정**: 탐색 키가 노드의 특정 키($k_i, k_{i+1}$) 사이의 범위에 있다면, 해당 분기점 사이의 자식 포인터를 따라 다음 레벨로 내려갑니다.


## 2-6. B-트리의 삽입 연산 (Insertion Operation)
B-트리의 삽입은 항상 **이파리 노드(Leaf Node)** 에서 발생하며, 노드의 크기를 적절히 유지하기 위해 **분리(Split)** 과정을 수반합니다.

### ① 삽입 위치 찾기  
* 탐색 연산과 동일한 과정을 거쳐 새로운 키가 저장될 적절한 리프 노드를 찾습니다.  

### ② 노드 수용 가능 여부 판단  
* **여유 공간이 있는 경우**: 리프 노드 내에서 키들이 정렬 상태를 유지하도록 적절한 위치에 새 키를 삽입하고 종료합니다.  
* **오버플로우(Overflow) 발생**: 리프 노드가 이미 **최대 키 수**($M-1$개) 를 가지고 있다면 **분할(Split)** 연산을 수행합니다.  

### ③ 분할(Split) 연산의 엄밀한 과정  
1. **중간 키 선출**: 기존 $M-1$개의 키와 새로 추가된 키를 합친 $M$개의 키 중 **중간값(Median)** 을 찾습니다.  
2. **부모로 승격**: 중간값 키를 부모 노드로 올려 보냅니다.  
3. **노드 분리**: 남은 $M-1$개의 키들을 중간값을 기준으로 절반씩 나누어($1/2$ 씩) 두 개의 별도 노드에 저장합니다.  
4. **연쇄적 분할**: 만약 중간값이 올라간 부모 노드 역시 이미 가득 차 있다면, 부모 노드에서도 동일하게 분할 연산을 수행합니다.
    이 과정이 루트까지 이어질 수 있습니다.  

## 2-7. 삽입 예제: 키 '45' 삽입  
<img width="1190" height="881" alt="image" src="https://github.com/user-attachments/assets/48cd450b-8161-4705-80ad-e91e9d010e03" />  

차수 $M=5$인 B-트리 예제를 분석합니다. (최대 키 수: 4개)  

1. **상태**: 리프 노드 $[25, 30, 35, 40]$에 키 **45**를 삽입하려 함.  
2. **오버플로우**: 키가 $[25, 30, 35, 40, 45]$가 되어 5개가 됨 (최대 4개 초과).  
3. **1차 분할**: 중간값 **35**를 부모 노드로 올림.   
   - 왼쪽 노드: $[25, 30]$  
   - 오른쪽 노드: $[40, 45]$  
4. **2차 분할 (부모 노드)**: 부모 노드가 $[5, 10, 20, 50]$이었으나 **35**가 올라오면서 $[5, 10, 20, 35, 50]$이 되어 다시 오버플로우 발생.  
5. **루트 생성**: 부모 노드의 중간값인 **20**을 다시 위로 올림. 20이 새로운 루트가 되며 트리의 높이가 1 증가함.  

## 2-8. B-트리의 삭제 연산 (Delete Operation)
B-트리의 삭제는 삽입보다 복잡하며, 트리의 균형을 유지하기 위해 항상 **이파리 노드(Leaf Node)**  에서 실질적인 삭제가 완료되도록 설계됩니다.

* **이파리 노드 삭제**: 키를 즉시 제거한 뒤, 노드의 최소 키 수 유지 여부를 확인합니다.   
* **내부 노드 삭제**:   
    - 이진 탐색 트리와 유사하게 **중위 선행자(In-order Predecessor)** 또는 **중위 후속자(In-order Successor)** 와 삭제할 키를 교환합니다.  
    - 교환 후, 이파리 노드로 내려온 키를 삭제합니다.  
* **언더플로우(Underflow)**: 삭제 후 노드의 키 수가 $\lceil M/2 \rceil - 1$보다 작아지면 균형이 깨진 상태(Underflow)로 간주하고 조치를 취합니다.  

## 2-9. 이동 연산 (Transfer)  
  
언더플로우가 발생했을 때, 인접한 **형제 노드의 키가 풍족한 경우** 사용합니다.  
  
* **조건**: 인접한 왼쪽 또는 오른쪽 형제 노드가 최소 키 수 이상의 키를 가지고 있어야 합니다.  
* **방법**: 형제 노드의 키 하나를 부모 노드로 올리고, 부모 노드에 있던 키를 언더플로우가 발생한 노드로 내려서 채웁니다. (부모를 거쳐서 빌려오는 방식)  
* **결과**: 트리의 높이는 변하지 않으며 지역적인 키 재배치로 해결됩니다.  


## 2-10. 통합 연산 (Fusion)  
  
형제 노드들이 모두 최소 키 수만 가지고 있어 **키를 빌려줄 수 없는 경우** 수행합니다.  
  
* **방법**: 언더플로우가 발생한 노드 $x$와 인접한 형제 노드를 **하나의 노드로 합칩니다.**  
* **부모 키의 강등**: 이때 두 노드의 분기점 역할을 하던 **부모 노드의 키를 통합된 노드로 끌어내려** 함께 저장합니다.  
* **연쇄 작용**: 부모 노드에서 키가 빠져나갔기 때문에, 부모 노드에서 다시 언더플로우가 발생할 수 있습니다. 이 과정은 루트까지 전파될 수 있습니다.

## 2-12. B-트리의 성능 분석
B-트리의 성능은 단순히 CPU 연산 횟수보다 **디스크 접근(Disk I/O) 횟수를 최소화**하는 데 초점이 맞춰져 있습니다.  

### ① 시간 복잡도와 트리의 높이  
* **연산 시간**: 삽입 및 삭제 연산의 수행 시간은 트리의 높이($h$)에 비례합니다.  
* **최대 높이**: 차수가 $M$이고 키의 개수가 $N$인 B-트리의 최대 높이는 수학적으로 다음과 같습니다.  
  $$h \le \log_{\lceil M/2 \rceil} \left( \frac{N+1}{2} \right) + 1 \implies O(\log_{M/2} N)$$  
* 이는 차수 $M$이 커질수록 트리의 높이가 급격히 낮아짐을 의미합니다.  

### ② 디스크 I/O 최적화 전략  
* **블록 이동(Transfer)의 최소화**: 컴퓨터 시스템에서 디스크와 메인 메모리 사이의 데이터 이동은 가장 비용이 많이 드는 작업입니다.  
  B-트리는 키 비교 횟수보다 **블록 이동 수**를 줄이는 것이 성능의 핵심입니다.    
* **노드 크기와 페이지 매칭**: 최적의 성능을 위해 **1개의 노드 크기가 디스크의 1개 페이지(Page) 또는 블록(Block) 크기**와  
  일치하도록 차수 $M$을 설정합니다. 이렇게 하면 한 번의 디스크 읽기로 하나의 노드 전체를 메모리로 가져올 수 있습니다.  

### ③ 압도적인 확장성 (실제 사례)  
실제 데이터베이스 시스템에서는 $M$의 크기를 수백에서 수천 단위로 설정합니다.  

* **예시 상황**: $M = 200$ (한 노드에 약 200개의 분기), $N = 1\text{억}$ (데이터 개수)    
* **결과**: 단 **4개의 디스크 블록**만 메인 메모리로 읽어 들이면 1억 개의 데이터 중 원하는 값을 찾아내거나 수정할 수 있습니다.  
  ($200^4 = 16\text{억}$이므로 높이 4 안에서 1억 개 수용 가능)  

### ④ 추가적인 최적화: 루트의 상주   
* **메인 메모리 상주**: 성능을 더욱 향상하기 위해, 탐색의 시작점인 **루트 노드는 항상 메인 메모리에 상주**시킵니다.  
  이를 통해 최소 한 번의 디스크 접근을 확정적으로 제거할 수 있습니다.
