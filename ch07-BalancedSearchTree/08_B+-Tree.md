## 3-1. B*-트리 (B-Star Tree)   
B*-트리는 B-트리의 공간 활용도를 더욱 높이기 위해 개선된 변형 구조입니다.  

* **핵심 차이점 (노드 점유율)**:   
    - B-트리는 노드의 최소 자식 수가 약 $1/2$($\lceil M/2 \rceil$)이지만, B*-트리는 이를 **$2/3$** 이상($\lceil 2/3M \rceil$)으로 강제합니다.  
* **오버플로우(Overflow) 처리**:   
    - 노드가 가득 찼을 때 즉시 분리(Split)하지 않고, 인접한 **형제 노드로 키를 재배치(Redistribution)** 합니다.  
    - 모든 형제 노드가 가득 찼을 때만 분리를 수행하여 분리 횟수를 줄입니다.
* **언더플로우(Underflow) 처리**:  
    - 키가 부족하면 인접 형제 노드에서 가져오며, 합병 시에는 3개의 노드를 2개로 합치는 방식을 사용하기도 합니다.  
* **장점**: B-트리에 비해 노드가 더 꽉 채워지므로 **공간 효율성**이 좋고 트리의 높이가 더 낮아질 가능성이 큽니다.

## 3-2. B+-트리 (B-Plus Tree)  
실세계의 **데이터베이스(DBMS) 및 파일 시스템**에서 가장 널리 사용되는 가장 중요한 B-트리 변형 모델입니다.  

### 1. 구조적 특징: 인덱스 노드와 데이터 노드  
B+-트리는 노드를 역할에 따라 두 종류로 명확히 구분합니다.  

| 구분 | 인덱스 노드 (Index Node) | 데이터 노드 (Data Node / Leaf) |  
| :--- | :--- | :--- |  
| **위치** | 루트 및 내부 노드 | 최하위 리프 노드 |  
| **저장 내용** | **탐색 키(Key)** 와 포인터만 저장 | **키 + 실제 데이터(Record)** 저장 |  
| **특징** | 길잡이 역할만 수행 (데이터 없음) | 모든 실제 정보가 이곳에 집결됨 |  
| **연결** | 트리 구조로 연결 | 리프 노드끼리 **연결 리스트**로 연결 |  

* **순차 탐색의 최적화**: 모든 리프 노드가 서로 링크(주로 이중 연결 리스트)되어 있어,  
  트리를 다시 올라가지 않고도 **범위 검색(Range Scan)** 이나 전체 레코드 순차 접근이 매우 빠릅니다.  
* **노드 크기**: 데이터 노드와 인덱스 노드의 크기는 같지 않아도 되며,  
 데이터 노드의 최소 원소 수는 대개 $\lceil c/2 \rceil$ (단, $c$는 데이터 노드의 최대 용량)입니다.  


### 2. B+-트리의 정의    
차수가 $m$인 B+-트리는 공백이거나 다음 성질을 만족해야 합니다.   
  
1.  **리프 노드의 정체성**: 모든 데이터 노드는 동일한 레벨에 위치해야 하며, 오직 리프 노드만이 실제 원소(Data)를 포함합니다.  
2.  **인덱스 노드의 제약**: 인덱스 노드는 차수가 $m$인 B-트리의 규칙을 따르지만, 오직 탐색을 돕는 키만 가질 뿐 실제 데이터를 가져서는 안 됩니다.  
3.  **형식적 명세**: 인덱스 노드는 $n, A_0, (K_1, A_1), \dots, (K_n, A_n)$ 형식을 가집니다.  
    - $A_i$: 서브트리에 대한 포인터  
    - $K_i$: 탐색 키 ($K_1 < K_2 < \dots < K_n$)  
    - 서브트리 $A_i$의 모든 키 값 $x$는 $K_i \le x < K_{i+1}$ 범위를 가집니다. (단, $K_0 = -\infty, K_{n+1} = \infty$)

## 3-3. B+-트리의 탐색 (Search)  

B+-트리는 구조적 특징 덕분에 두 가지 강력한 탐색 방식을 지원합니다.  

* **탐색 종류**:  
    1. **정확히 일치하는 값 검색 (Point Query)**: 특정 키를 가진 레코드를 탐색.  
    2. **범위 검색 (Range Query)**: 특정 범위($k_1 \le x \le k_2$) 내의 모든 레코드를 탐색. 리프 노드의 연결 리스트를 따라가며 매우 빠르게 수행됩니다.  
* **탐색 알고리즘 로직**:  
    1. **루트 노드**에서 시작하여 현재 노드가 **인덱스 노드**인 동안 반복합니다.  
    2. 인덱스 노드의 형식 $n, A_0, (K_1, A_1), \dots, (K_n, A_n)$을 확인하여, 탐색 키 $x$가 $K_i \le x < K_{i+1}$ 범위를 만족하는  
       포인터 $A_i$를 선택해 아래로 내려갑니다.  
    3. 최하위 **데이터 노드(리프)** 에 도달하면, 해당 노드 내에서 키 $x$를 가진 원소 $E$를 탐색하여 있으면 반환하고 없으면 NULL을 반환합니다.  

## 3-4. B+-트리의 삽입 (Insertion)  
B+-트리의 삽입은 데이터 노드(리프)가 분할될 때 부모로 키가 전달되는 방식이 B-트리와 다릅니다.  

### ① 데이터 노드(Leaf) 분할  
* 데이터 노드가 가득 차면(Overflow), 원소들을 정렬한 후 **절반을 새로운 노드로** 옮깁니다.  
* **키의 복사**: 새 데이터 노드의 **가장 작은 키**를 부모 인덱스 노드로 복사하여 올립니다. (원래 데이터는 리프 노드에 그대로 남아있어야 하기 때문입니다.)  

### ② 인덱스 노드(Internal) 분할  
* 내부 노드인 인덱스 노드가 가득 차면, 기존 **B-트리의 내부 노드 분할 방식과 동일**하게 중간 키가 부모로 올라가며 분할됩니다.  

### ③ 삽입 예제 분석    
* **14 삽입**: 리프 노드 B에 빈 자리가 있어 12와 16 사이에 정렬되어 들어갑니다.

## 3-5. B+-트리의 삭제 (Deletion)  
B+-트리에서 모든 실제 원소는 리프 노드에만 존재하므로, **리프 노드에서의 삭제 연산**이 핵심입니다.  
  
### ① 최소 원소 수 부족(Underflow) 기준  
* **인덱스 노드**: 루트가 아닌 경우, 키의 개수가 $\lceil m/2 \rceil - 1$ 개 미만이 될 때.  
* **데이터 노드**: 루트가 아닌 경우, 원소의 수가 $\lceil c/2 \rceil$ 개 미만이 될 때. ($c$: 데이터 노드의 최대 용량)  

### ② 삭제 후 조정 과정  
1. **Underflow가 발생하지 않은 경우**: 데이터 노드에서 원소만 삭제하고 디스크에 기록합니다. (인덱스 노드는 변경되지 않음)  
2. **Underflow가 발생한 경우**:  
    * **빌려오기(Borrowing)**: 인접한 형제 노드에 여유 원소가 있다면 하나를 가져오고,  
       부모(인덱스 노드)의 해당 키 값을 새 리프의 최소 키 값으로 갱신합니다.  
    * **병합(Merging)**: 형제에게 빌릴 수 없다면 형제 노드와 하나로 합치고, 부모 노드에서 해당 분기 키를 삭제하거나 조정합니다.  
       이로 인해 부모 노드에 연쇄적인 Underflow가 발생할 수 있습니다.  

### ③ 삭제 예제 분석  
* **71, 80 삭제**: 리프에서 제거. 71 삭제 시 이를 가이드하던 인덱스 노드의 키가 72로 자동 갱신됩니다.

## 4. 요약  
  
지금까지 배운 균형 탐색 트리들의 핵심 개념을 정리합니다.  

| 트리 종류 | 핵심 정의 및 특징 | 수행 시간 |  
| :--- | :--- | :---: |  
| **2-3 트리** | 모든 내부 노드의 차수가 2 또는 3인 완전 균형 트리. | $O(\log N)$ |  
| **2-3-4 트리** | 4-노드까지 허용. 루트에서 내려가며 **미리 분리/통합**하여 효율적임. | $O(\log N)$ |  
| **B-트리** | 다방향 탐색 트리. **디스크 페이지 단위** 접근에 최적화된 $M$차 트리. | $O(\log N)$ |  
| **B\*-트리** | 노드 점유율을 $2/3$ ( $67\%$ ) 이상으로 강제하여 공간 효율 극대화. | $O(\log N)$ |  
| **B+-트리** | **인덱스 노드**와 **데이터 노드**를 분리. 리프 연결 리스트로 **범위 탐색**에 최적화. | $O(\log N)$ |
