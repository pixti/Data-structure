## 1. 우선순위 큐(Priority Queue)의 필요성

## 1-1. 정적(Static) vs 동적(Dynamic) 데이터 집합
* **정적 데이터 집합:** 데이터가 한 번 구축되고 나면 변하지 않습니다. 데이터가 들어오고 나가는 일이 없으므로, 처음에 한 번 정렬해 두면 이후 작업이 매우 단순합니다.
* **동적 데이터 집합:** 데이터가 수시로 삽입, 삭제, 변경됩니다. 실제 현실의 작업(운영체제, 네트워크 등)은 대부분 동적 데이터이며, 매번 정렬을 새로 하는 것은 비효율적입니다.

## 1-2. 특정 우선순위만 알면 되는 CPU 작업
* 운영체제의 CPU 스케줄링처럼 작업 간 우선순위가 존재할 때, 우리는 **우선순위가 가장 높은 녀석 하나**만 알면 됩니다.
* 굳이 모든 데이터의 위치를 파악하거나 특정 우선순위를 검색해서 찾아낼 필요가 없습니다. 즉, "최고의 작업" 하나만 빠르게 뽑아내는 것이 핵심입니다.

## 1-3. 일반적인 배열(선형 구조)의 문제점
데이터를 정렬해서 보관하려고 할 때, 오름차순이든 내림차순이든 배열은 한계가 명확합니다.
* **데이터 삽입/삭제 시:** 새로운 데이터가 들어오거나 나갈 때마다 다른 데이터들을 옆으로 밀어내거나 당겨야 하는 **Shifting(밀어내기)** 문제가 발생합니다.
* **성능:** 데이터가 많아질수록 이 밀어내는 작업( $O(N)$ )이 전체 성능의 발목을 잡게 됩니다.

## 2. 우선순위 큐의 정의 및 특징

## 2-1. 우선순위 큐(Priority Queue)란?
가장 높은 우선순위를 가진 항목에 접근하여 삭제하고, 새로운 항목을 임의의 우선순위와 함께 삽입하는 자료구조입니다.

## 2-2. 스택과 큐의 재해석
사실 우리가 배운 기본 자료구조들도 '시간'을 우선순위로 둔 일종의 우선순위 큐로 볼 수 있습니다.
* **스택(Stack):** 가장 마지막에 들어온 데이터에 가장 높은 우선순위를 부여 (LIFO).
* **큐(Queue):** 가장 먼저 삽입된 데이터에 가장 높은 우선순위를 부여 (FIFO).

## 2-3. 데이터 중복 허용
일반적인 Dictionary(Table)는 Key의 중복을 허용하지 않지만, **우선순위 큐는 동일한 우선순위 값을 가진 데이터가 여러 개 저장될 수 있습니다.** 작업 스케줄링에서 우선순위가 같은 작업이 여러 개 있을 수 있는 것과 같은 원리입니다.

## 3. 우선순위 큐의 ADT (추상 자료형)

우선순위 큐는 "무엇을 하는가(What)"를 정의하는 ADT 단계에서 다음과 같은 기능을 명세합니다.

* **insert(k, v):** 우선순위 `k`와 값 `v`를 가진 항목을 삽입.
* **min():** 가장 작은 우선순위(높은 우선순위)를 가진 항목을 반환.
* **removeMin():** 가장 작은 우선순위를 가진 항목을 반환하며 삭제.
* **size() / isEmpty():** 항목 수 확인 및 비어있는지 체크.

## 4. 구현 방식에 따른 성능 비교

| 연산 (Method) | 정렬되지 않은 리스트 (Unsorted) | 정렬된 리스트 (Sorted) |
| :---: | :---: | :---: |
| **insert (삽입)** | $O(1)$ | $O(n)$ |
| **min (최소값 탐색)** | $O(n)$ | $O(1)$ |
| **removeMin (삭제)** | $O(n)$ | $O(1)$ |

## 4-1. 정렬되지 않은 리스트의 한계
* 삽입은 아무 데나 넣으면 되니 빠르지만( $O(1)$ ), 가장 작은 값을 찾으려면 전체를 다 뒤져야 하므로 $O(n)$ 이 걸립니다.

## 4-2. 정렬된 리스트의 한계
* 가장 작은 값이 맨 앞에 있으니 찾는 건 빠르지만( $O(1)$ ), 새로운 값을 넣을 때 순서를 맞추기 위해 데이터를 밀어내야 하므로 $O(n)$이 소요됩니다.

## 5. 대체 기술: 트리(Tree)와 힙(Heap)

선형 구조(배열/리스트)의 한계를 극복하기 위해 계층 구조인 트리를 도입합니다.

## 5-1. 이진 탐색 트리 (BST)
* 삽입과 최소값 탐색을 모두 **$O(\log N)$** 수준으로 처리할 수 있어 선형 구조보다 훨씬 유리합니다.

## 5-2. 힙 (Heap)
* 배열의 물리적 이점을 살리면서도 트리 구조를 유지하는 방식입니다.
* 가장 우선순위가 높은 값을 항상 루트(Root)에 두어 **찾는 것은 $O(1)$**, **지우고 다시 구조를 잡는 것은 $O(\log N)$** 만에 해결하여 효율성을 극대화합니다.

## 6. 힙(Heap): 대표적인 우선순위 큐

## 6-1. 힙(Heap)이란?
우선순위를 효율적으로 관리하기 위해 고안된 '특수한 성질을 가진 이진 트리'입니다.
* **주의:** 자바(Java) 등 프로그래밍 언어에서 말하는 메모리 공간인 '힙(Heap) 영역'과는 이름만 같을 뿐 전혀 다른 개념입니다.

## 6-2. 힙이 되기 위한 2가지 필수 조건
어떤 트리가 '힙'이라고 불리려면 다음 두 가지 조건을 **모두** 만족해야 합니다.

### ① 완전 이진 트리 (Complete Binary Tree)
* 트리의 노드가 위에서 아래로, 왼쪽에서 오른쪽으로 빈틈없이 차곡차곡 채워진 구조여야 합니다.
* 트리의 중간에 **'구멍'이 나 있다면 완전 이진 트리가 아니며**, 따라서 힙도 아닙니다.
* **성능적 이점:** AVL 트리나 레드-블랙 트리 같은 이진 탐색 트리들은 좌우 균형을 맞추기 위해 복잡한 회전 연산이 필요하여 체감 오버헤드가 있지만, 힙은 단순히 '완전 이진 트리'라는 형태만 배열로 꽉 채워 유지하면 되므로 구조 유지가 훨씬 단순하고 빠릅니다.

### ② 힙 특성 (Heap Property)
* **부모 노드의 우선순위는 항상 자식 노드의 우선순위보다 높거나 같아야 합니다.**
* 형제 노드(왼쪽 자식과 오른쪽 자식) 사이의 대소 관계는 전혀 따지지 않습니다. (왼쪽이 크든 오른쪽이 크든 무관함)
* 이 특성 덕분에 **루트(Root) 노드는 항상 트리 전체에서 가장 우선순위가 높은 값**을 가지게 됩니다.

## 6-3. 최대힙(Max Heap)과 최소힙(Min Heap)
우선순위의 기준(크다/작다)에 따라 두 가지로 나뉩니다.

* **최대힙 (Max Heap):** 값이 큰 것이 우선순위가 높다고 정의한 힙입니다. 부모 노드의 값이 자식 노드보다 항상 크거나 같으며, 루트 노드에 **최댓값**이 위치합니다.
* **최소힙 (Min Heap):** 값이 작은 것이 우선순위가 높다고 정의한 힙입니다. 부모 노드의 값이 자식 노드보다 항상 작거나 같으며, 루트 노드에 **최솟값**이 위치합니다.

> **참고:** 최대힙과 최소힙은 부등호 방향(비교 기준)만 다를 뿐 구조와 원리는 완전히 대칭적입니다. 따라서 하나만 제대로 이해하면 값의 대소만 변경하여 다른 하나도 똑같이 구현할 수 있습니다.

## 7. 이진 힙(Binary Heap)의 배열 구현 원리

## 7-1. 완전 이진 트리와 1차원 배열의 완벽한 조화
**이진 탐색 트리(BST)** 는 한쪽으로 치우치는(Skewed) 현상이 발생할 수 있어, 이를 배열로 구현하면 빈 공간(구멍)이 무수히 생겨 메모리 낭비가 심합니다. 

하지만 **힙(Heap)은 무조건 '완전 이진 트리'** 구조를 유지한다는 엄격한 조건이 있습니다. 따라서 배열에 빈틈없이 차곡차곡 데이터를 채워 넣을 수 있으므로, 복잡한 포인터(Link) 구조 없이 **단순한 1차원 배열로 구현하는 것이 가장 효율적이고 이상적입니다.**
* **저장 방식:** 트리의 노드들을 **레벨 순회(Level-order Traversal)** 순서대로, 즉 위에서 아래로, 왼쪽에서 오른쪽 순서로 배열에 차례대로 저장합니다.

## 7-2. 인덱스 계산 공식 (부모-자식 관계 찾기)
배열을 이용하면 부모와 자식 노드의 물리적 위치를 단순한 사칙연산만으로 즉시 알아낼 수 있습니다. 배열의 0번 인덱스 사용 여부에 따라 공식이 나뉩니다.

### ① 1번 인덱스부터 시작하는 경우 (표준적이고 직관적인 방법)
* `A[0]`은 비워두고, 루트 노드를 `A[1]`에 저장합니다. 곱셈과 나눗셈이 깔끔하게 떨어져 알고리즘 설명이나 수학적 계산에서 가장 선호되는 방식입니다. 특정 노드의 인덱스가 $i$일 때:
    * **왼쪽 자식 위치:** $2i$
    * **오른쪽 자식 위치:** $2i + 1$
    * **부모 노드 위치:** $\lfloor i / 2 \rfloor$ (소수점 버림)

### ② 0번 인덱스부터 시작하는 경우 (실제 프로그래밍 언어 구현)
* 배열의 공간 낭비를 1칸도 허용하지 않기 위해 루트 노드를 `A[0]`에 저장합니다. 공식이 조금 변형됩니다. 특정 노드의 인덱스가 $i$일 때:
    * **왼쪽 자식 위치:** $2i + 1$
    * **오른쪽 자식 위치:** $2i + 2$
    * **부모 노드 위치:** $\lfloor (i - 1) / 2 \rfloor$

## 7-3. 루트(Root) 노드의 접근 속도
* 배열의 첫 번째 칸(인덱스 `1` 혹은 `0`)은 항상 트리의 가장 꼭대기인 루트 노드를 가리킵니다.
* **최소힙(Min Heap)** 이라면 배열의 첫 번째 칸에 항상 트리 전체의 **최솟값** 이 위치하며, **최대힙(Max Heap)** 이라면 **최댓값** 이 위치합니다.
* 따라서 현재 큐에서 가장 우선순위가 높은(최대 또는 최소) 값을 단순히 들여다보는 작업(`min()` 또는 `max()`)은 배열의 첫 번째 인덱스만 읽으면 되므로 항상 **$O(1)$** 의 성능을 보장합니다.

## 8. 힙 객체 구조 및 삽입(Insert) 연산

## 8-1. 힙의 객체 구조
힙은 복잡한 트리 노드(Node) 객체나 포인터(Link) 없이, 오직 **1차원 배열**과 **항목의 개수를 추적하는 변수** 만으로 완벽하게 구현됩니다.

* **`A[]`:** 힙의 원소들이 실제 저장되는 배열입니다.
* **`numItems`:** 현재 힙에 저장된 실제 데이터의 개수입니다. 배열의 전체 길이(`length`)와는 다르며, 삽입/삭제 시 이 값을 기준으로 트리의 마지막 위치를 파악합니다.

## 8-2. 삽입 연산의 핵심 딜레마: "어디에 넣을 것인가?"
새로운 데이터를 힙에 추가할 때, 무작정 배열의 앞쪽(루트나 중간)에 끼워 넣으려고 하면 기존 데이터들을 모두 뒤로 밀어내야(Shifting) 하는 $O(N)$의 비용이 발생합니다. 

* **해결책:** 밀어내기 연산을 피하고 **'완전 이진 트리'의 형태를 무조건 유지** 하기 위해, 새로운 데이터는 항상 트리의 **가장 마지막 빈자리(배열의 맨 끝, `A[numItems]`)** 에 일단 집어넣습니다.

## 8-3. 상향식 재구조화: Upheap (또는 PercolateUp)
일단 맨 끝에 데이터를 넣으면 완전 이진 트리의 형태는 유지되지만, 부모-자식 간의 크기 관계를 따지는 **힙 특성(Heap Property)이 깨질 수 있습니다.** 이를 복구하기 위해 **밑에서 위로(Bottom-Up)** 올라가며 자리를 교환합니다.

### 삽입 및 재구조화 과정 (최대힙 기준)
1. **맨 끝 삽입:** 새로운 원소를 배열의 맨 마지막 원소 바로 다음 빈칸에 저장합니다.
2. **부모와 비교:** 루트 방향으로 올라가면서, 새로 삽입된 노드의 키 값과 그 부모 노드의 키 값을 비교합니다.
    * *참고:* 배열 인덱스가 $i$일 때 부모의 인덱스는 $\lfloor (i-1)/2 \rfloor$ 또는 $\lfloor i/2 \rfloor$ (시작 인덱스에 따라 다름)로 쉽게 찾을 수 있습니다.
3. **교환(Swap):** 자식(새 데이터)의 값이 부모보다 크다면 힙 특성에 위배되므로 두 노드의 위치를 맞바꿉니다. (사용자 메모의 표현처럼, 부모를 아래로 끌어내리고 새 데이터를 위로 올리는 과정입니다.)
4. **종료 조건:** 부모가 자신보다 크거나 같아질 때, 혹은 트리 구조의 최상단인 루트 노드에 도달할 때까지 2~3번 과정을 반복(재귀)합니다.

## 8-4. 시간 복잡도
* 힙은 완전 이진 트리이므로 데이터가 $N$개일 때 트리의 높이는 항상 $\log N$입니다.
* 새로운 데이터가 루트까지 거슬러 올라가는 최악의 경우(Worst Case)라도 트리의 높이만큼만 교환이 일어나므로, 삽입 연산의 시간 복잡도는 항상 **$O(\log N)$**을 보장합니다.

## 8-5. 삽입 알고리즘(Insert) 로직 비교

배열 기반의 힙 삽입 연산은 `while`문을 이용한 반복문 방식과, 부모 노드를 인자로 넘겨가며 처리하는 재귀(Recursion) 방식으로 모두 구현할 수 있습니다. (아래는 0번 인덱스를 사용하는 힙 기준의 수도코드 형태입니다.)

### ① 반복문 (Iterative) 방식
> insert(x):
>     i = n          // n은 현재 원소 수 (삽입될 새 위치)
>     A[i] = x       // 맨 끝에 원소 삽입
>     parent = (i-1) / 2
>     while (i > 0 and A[i] > A[parent]):
>         A[i] ↔ A[parent]  // 부모와 맞바꾸기
>         i = parent
>         parent = (i-1) / 2
>     n++            // 힙 크기 1 증가

### ② 재귀 (Recursive) 방식
서브 트리에서 위치가 맞지 않으면 부모와 교환하고, 다시 부모의 입장에서 구조화 작업을 반복하는 특성을 살려 깔끔하게 구현할 수 있습니다.
> insert(x):
>     A[n] = x       // 맨 끝에 삽입
>     percolateUp(n) // 재구조화 함수 호출
>     n++
>
> percolateUp(i):
>     parent = (i-1) / 2
>     if (i > 0 and A[i] > A[parent]):
>         A[i] ↔ A[parent]      // 맞바꾸기
>         percolateUp(parent) // 바뀐 위치에서 다시 재귀 호출
