## 1. 우선순위 큐(Priority Queue)의 필요성 및 개념

## 1-1. 왜 우선순위 큐가 필요한가? (동적 데이터 처리)
* **정적(Static) 데이터:** 한 번 구축되면 변하지 않는 데이터 집합입니다. 처음에 한 번만 정렬해 두면 탐색이나 우선순위 파악이 매우 쉽습니다.
* **동적(Dynamic) 데이터:** 데이터가 계속해서 추가되거나 삭제, 변경되는 집합입니다. 매번 데이터가 들어올 때마다 전체를 다시 정렬하는 것은 성능상 비효율적입니다.
* **결론:** 데이터가 수시로 변하는 동적 환경에서, "가장 우선순위가 높은 데이터"를 빠르게 찾고 삭제하기 위해 고안된 자료구조가 바로 우선순위 큐입니다.

## 1-2. 우선순위 큐의 핵심 개념
우선순위 큐는 **임의의 우선순위를 가진 항목을 삽입**하고, **가장 높은 우선순위를 가진 항목에 접근 및 삭제**하는 것을 지원하는 자료구조입니다.

* **CPU 작업 스케줄링 예시:** 운영체제의 CPU 스케줄러는 수많은 작업 중 '가장 우선순위가 높은 작업 1개'만 알면 됩니다. 특정 우선순위를 가진 다른 중간 작업을 굳이 검색해서 찾을 필요가 없습니다. 
* **스택과 큐도 일종의 우선순위 큐:**
    * **스택(Stack):** '가장 마지막으로 삽입된 시간'이 가장 높은 우선순위가 되는 구조 (LIFO).
    * **큐(Queue):** '가장 먼저 삽입된 시간'이 가장 높은 우선순위가 되는 구조 (FIFO).

## 1-3. Dictionary(Table) vs 우선순위 큐 비교
둘 다 동적 데이터를 다루지만, 목적과 허용 규칙이 다릅니다.

| 기능 | Dictionary (Hash Table 등) | 우선순위 큐 (Priority Queue) |
| :--- | :--- | :--- |
| **삭제 기준** | 특정 Key를 기반으로 임의의 원소 삭제 | **오직 우선순위가 가장 높은 원소만** 삭제 |
| **삽입** | 임의의 원소 삽입 가능 | 임의의 원소 삽입 가능 |
| **중복 허용 여부** | Key의 중복을 **불허** | 같은 우선순위 값의 중복을 **허용** |
| **특정 원소 검색** | 지원함 (빠른 검색이 목적) | 지원하지 않음 (최고 우선순위만 알면 됨) |

## 2. 우선순위 큐의 ADT (추상 자료형)

우선순위 큐가 기본적으로 제공해야 하는 작업(기능)의 명세입니다. (Min-Priority Queue 기준)

* **`insert(k, v)`:** 키 $k$(우선순위)와 값 $v$를 가진 원소를 삽입합니다.
* **`min()`:** 가장 우선순위가 높은(값이 작은) 원소를 알려줍니다. (삭제하지 않음)
* **`removeMin()`:** 가장 우선순위가 높은 원소를 알려주면서 동시에 큐에서 **삭제**합니다.
* **`size()` / `isEmpty()`:** 큐의 크기를 반환하거나 비어있는지 확인합니다.


## 3. 구현 방식에 따른 성능(시간 복잡도) 한계점

우선순위 큐를 단순한 리스트(배열)로 구현했을 때의 한계점과 시간 복잡도를 비교합니다.

| 연산 (Method) | 정렬되지 않은 리스트 (Unsorted) | 정렬된 리스트 (Sorted) |
| :---: | :---: | :---: |
| **size / isEmpty** | $O(1)$ | $O(1)$ |
| **insert (삽입)** | **$O(1)$** | **$O(N)$** |
| **min (최소값 확인)** | $O(N)$ | $O(1)$ |
| **removeMin (최소값 삭제)** | **$O(N)$** | **$O(1)$** |

### ① 정렬되지 않은 리스트의 문제점
* **삽입:** 맨 뒤에 그냥 넣으면 되므로 $O(1)$입니다.
* **삭제/탐색:** 제일 우선순위가 높은 것을 찾으려면 처음부터 끝까지 다 뒤져봐야(순차 탐색) 하므로 **$O(N)$** 이 소요됩니다.

### ② 정렬된 배열의 문제점
* **삭제/탐색:** 항상 정렬되어 있으니 맨 앞(또는 맨 뒤)의 값을 빼기만 하면 되어 $O(1)$입니다.
* **삽입:** 오름차순이든 내림차순이든, 중간에 데이터를 끼워 넣으려면 기존 데이터들을 모두 뒤로 밀어내야(Shifting) 하므로 **$O(N)$** 이 소요되어 비효율적입니다.

### ③ 해결책: 트리(Tree) 구조의 도입
배열의 밀어내기 문제( $O(N)$ )와 순차 탐색 문제( $O(N)$ )를 모두 극복하기 위해 등장한 대안입니다.
* **이진 탐색 트리 (BST):** 삽입과 삭제, 최소값 탐색을 모두 $O(\log N)$ 에 처리할 수 있습니다.
* **힙 (Heap):** 오직 최고 우선순위를 찾는 데 특화된 트리로, 배열 형태를 유지하면서도 **최소값 확인은 $O(1)$, 삽입과 삭제는 $O(\log N)$** 이라는 최적의 성능을 보장합니다. (우선순위 큐의 실질적인 표준 구현체)


## 4. 자바(Java) ADT 인터페이스 구조

자바에서 우선순위 큐를 구현하기 위한 기본 골격(Interface)과 키-값 쌍(Entry) 구조입니다.

### ① Entry (키-값 쌍) 인터페이스
우선순위를 결정하는 `Key`와 실제 데이터인 `Value`를 하나로 묶어 관리합니다.

> public interface Entry<K, V> {
>     K getKey();   // 엔트리에 저장된 키(우선순위) 반환
>     V getValue(); // 엔트리에 저장된 값 반환
> }

### ② PriorityQueue 인터페이스 명세
우선순위 큐가 수행해야 할 필수 기능들을 선언합니다.

> public interface PriorityQueue<K, V> {
>     int size();
>     boolean isEmpty();
>     Entry<K, V> insert(K key, V value) throws IllegalArgumentException;
>     Entry<K, V> min();
>     Entry<K, V> removeMin();
> }
