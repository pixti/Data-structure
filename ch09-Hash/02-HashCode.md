## 1. 자바(Java)의 hashCode()

## 1-1. hashCode()란?
자바의 모든 클래스는 객체를 32비트 정수(`int`)로 변환해 반환하는 `hashCode()` 메서드를 기본적으로 포함하고 있습니다. **즉, 어떤 복잡한 형태의 데이터(객체)라도 궁극적으로 해시 테이블에서 사용할 수 있도록 하나의 '정수(int)' 값으로 바꿔주는 것이 가장 중요하고 핵심적인 역할입니다.**

### 핵심 규칙 (Contract)
* `key1.equals(key2)`가 `true`라면, `key1.hashCode() == key2.hashCode()` 역시 반드시 성립해야 합니다.
* 즉, 논리적으로 동일한 두 객체는 무조건 같은 해시코드 값을 가져야 합니다.

## 1-2. 기본 래퍼(Wrapper) 클래스의 hashCode() 구현
각 자료형의 데이터 특성에 맞춰 해시값을 도출하는 최적의 방식이 구현되어 있습니다.

### ① Integer 객체
* 별도의 해시 연산 과정 없이, 자신이 가진 정수값(`key`) 자체를 그대로 반환합니다.

> public int hashCode() {
>     return key;
> }

### ② Boolean 객체
* 값이 `true`일 경우 **1231**, `false`일 경우 **1237**이라는 고정된 소수(Prime number)를 반환합니다.

> public int hashCode() {
>     if (key) return 1231;
>     else return 1237;
> }

### ③ Double 객체
* 64비트(8바이트)의 부동소수점 데이터를 32비트(4바이트) 크기의 `int`로 압축하기 위해 **비트 연산(XOR)** 을 사용합니다.
* 데이터를 IEEE 64-bit 포맷으로 변환한 뒤, **최상위 32비트와 최하위 32비트를 XOR(`^`) 연산**합니다. 데이터의 모든 비트를 고르게 참여시켜 충돌을 줄이기 위함입니다.
* **성능적 이점:** 복잡한 수학 연산이 아니라 하드웨어 수준에서 처리되는 **비트(Bit) 연산자**를 사용하기 때문에, 연산 속도가 매우 빠르고 처리에 걸리는 시간이 거의 없다는 큰 장점이 있습니다.

> public int hashCode() {
>     long bits = doubleToLongBits(key);
>     return (int) (bits ^ (bits >>> 32));
> }

## 1-3. String 객체의 hashCode() 구현 원리
문자열은 내부 문자 배열의 각 문자(`char`)가 가진 **아스키코드(ASCII) 혹은 유니코드(Unicode) 정수값**을 바탕으로, 이를 **31진법 체계의 숫자**로 보고 다항식 형태로 해시값을 계산합니다.

### 계산 공식
문자열의 길이가 $n$일 때, $i$번째 문자 $s[i]$에 대하여 다음과 같이 계산합니다.
$$hash = \sum_{i=0}^{n-1} s[i] \cdot 31^{n-1-i}$$

### 예시: "ball"의 해시값 계산
각 문자의 유니코드 값: `b=98`, `a=97`, `l=108`
* $hash = 98 \cdot 31^3 + 97 \cdot 31^2 + 108 \cdot 31^1 + 108 \cdot 31^0$
* 루프를 돌며 최적화된 연산: $108 + 31 \cdot (108 + 31 \cdot (97 + 31 \cdot (98))) = 3016191$

> public int hashCode() {
>     int hash = 0;
>     for (int i = 0; i < length(); i++)
>         hash = s[i] + (31 * hash);
>     return hash;
> }

## 1-4. 해시 테이블(Hash Table) 적용 시 주의사항
자바를 이용해 해시 테이블을 직접 구현할 때, 객체의 `hashCode()` 반환값을 곧바로 배열의 인덱스(Index)로 사용할 수 없습니다. 반환값이 음수일 수도 있고, 해시 테이블의 크기를 초과할 수도 있기 때문입니다.

### 안전한 인덱스 변환 과정
1. **음수 방지 (비트 마스킹):** 반환된 32비트 정수의 부호 비트(최상위 비트)를 0으로 지우기 위해 `0x7fffffff`와 비트 AND(`&`) 연산을 수행하여 무조건 양수로 만듭니다. (이 역시 비트 연산이라 매우 빠릅니다.)
2. **크기 맞춤 (모듈러 연산):** 얻어진 양수를 해시 테이블의 크기($M$)로 나눈 나머지(`% M`)를 최종 인덱스로 사용합니다.

> private int hash(Key k) {
>     return (k.hashCode() & 0x7fffffff) % M;
> }

## 2. 해시 테이블의 충돌(Collision)과 해결 방법

## 2-1. 충돌(Collision)이란?
* 아무리 우수한 해시 함수를 사용하더라도, 서로 다른 키(Key)들이 동일한 해시값을 가져 해시 테이블의 **같은 원소(인덱스)에 배정되어야 하는 상황**이 필연적으로 발생합니다. 이를 **충돌(Collision)** 이라고 합니다.
* 비유하자면, 서로 다른 과일(포도, 체리, 라임)이 모두 5번 칸에 들어가려고 배정받은 상황과 같습니다.

## 2-2. 충돌 해결(Collision-Resolution) 방법
해시 테이블은 한 칸에 하나의 데이터만 들어갈 수 있는 배열을 기반으로 하므로, 충돌이 발생했을 때 이를 해결할 전략이 필요합니다. 크게 **개방 주소 방법**과 **폐쇄 주소 방법**으로 나뉩니다.


## 2-3. 개방 주소 방법 (Open Addressing) : "다른 칸에 넣을게요"
해시 테이블 전체를 열린 공간으로 가정하고, 내 자리에 이미 데이터가 있다면 일정한 규칙에 따라 **빈 공간(Empty)을 찾아 다른 원소에 저장**하는 방식입니다. 오직 주어진 배열 안에서만 문제를 해결합니다.

### ① 선형 탐색 (Linear Probing) : "다음 칸, 다음 칸, 다음 칸..."
* 충돌이 발생하면 바로 그 다음 칸을 확인하고, 거기도 차있으면 또 다음 칸을 확인하는 가장 단순한 방식입니다.
* **수식:** $h_i(x) = (h_0(x) + ai + b) \pmod m$ ($m$은 테이블 사이즈)
* **단점:** 특정 영역에 데이터가 몰리는 군집화(Clustering) 현상이 발생하여 탐색 시간이 길어질 수 있습니다.

### ② 이차원 탐색 (Quadratic Probing) : "제곱으로 건너뛰기"
* 선형 탐색의 군집화 문제를 완화하기 위해, 다음 빈 칸을 찾을 때 보폭을 $1^2, 2^2, 3^2, ...$ 처럼 제곱수로 늘려가며 탐색합니다.
* **수식:** $h_i(x) = (h_0(x) + ai^2 + bi + c) \pmod m$

### ③ 더블 해싱 (Double Hashing) : "이중 해싱"
* 충돌이 발생했을 때 이동할 보폭을 아예 **별도의 보조 해시 함수($h_0'(x)$)**를 사용하여 결정합니다. 데이터마다 이동 폭이 달라지므로 군집화 현상을 가장 잘 방지합니다.
* **수식:** $h_i(x) = (h_0(x) + i \cdot h_0'(x)) \pmod m$

### ④ 랜덤 조사 (Random Probing) : "무작위로 점프하기"
* 충돌이 발생하면 의사 난수 발생기(Pseudo-Random Number Generator)를 사용하여 무작위로 생성된 난수 수열에 따라 다음으로 이동할 빈 칸을 결정하는 방식입니다.
* **장점:** 이동 보폭에 규칙적인 패턴이 없으므로, 데이터가 특정 영역에 뭉치는 군집화 현상을 효과적으로 분산시킬 수 있습니다.


## 2-4. 폐쇄 주소 방법 (Closed Addressing) : "내 칸에 무조건 넣을게요"
충돌이 발생해도 다른 칸으로 도망가지 않고, **원래 배정받은 자기 자리에서 해결**하는 방식입니다. 

### ① 체이닝 (Chaining)
* 같은 자리에서 충돌이 일어난 원소들을 **연결 리스트(Linked List)** 로 주렁주렁 매달아 관리하는 방식입니다.
* 테이블의 각 원소(`table[i]`)는 실제 데이터를 가지는 것이 아니라, 해당 인덱스에 배정된 **연결 리스트의 첫 번째 노드를 가리키는 링크(참조)** 를 가집니다.
* **장점:** 배열이 꽉 차는 100% 충돌 상황에서도 연결 리스트만 길어질 뿐 계속 저장이 가능합니다. (다만 검색 속도는 느려집니다.)


## 2-5. 해시 테이블의 동적 확장 (Resizing)
* **적재율(Load Factor):** 해시 테이블에 데이터가 얼마나 차 있는지를 나타내는 비율입니다. (데이터 개수 / 테이블 크기)
* 체이닝을 쓰든 개방 주소법을 쓰든, 데이터가 많아져 적재율이 높아지면(통상적으로 **65% ~ 75%** 이상) 충돌이 급격히 잦아지고 성능이 저하됩니다.
* **해결책:** 이 임계치를 넘으면, **내부 배열의 크기를 2배로 늘리고** 기존의 모든 데이터를 새로운 크기에 맞춰 다시 해싱(Rehashing)하여 재배치합니다. ArrayList의 동적 확장(Resizing) 원리와 동일합니다.

## 3. 개방 주소 방식: 선형 조사 (Linear Probing)

## 3-1. 선형 조사(Linear Probing)란?
선형 조사는 개방 주소법 중 가장 단순한 형태로, 충돌이 발생하면 **순차적으로 다음 빈 칸을 검색하여 데이터를 저장**하는 방식입니다.

* **핵심 원리:** "내 원래 자리에 누군가 있다면, 바로 다음 칸에 넣는다."
* **배열과의 차이점:** 일반적인 배열의 삽입처럼 중간에 데이터를 넣기 위해 기존 데이터들을 뒤로 밀어내지(Shifting) 않습니다. 단순히 다음 빈 인덱스를 찾아 빈자리에 '쏙' 집어넣기만 합니다.
* **원형 탐색 (Wrap-around):** 탐색하다가 배열의 끝 인덱스에 도달하면, 원형 큐(Circular Queue)처럼 다시 배열의 맨 처음(인덱스 0)으로 돌아와서 탐색을 계속합니다.
  * **수식:** $i = (initialpos + j) \pmod M$ (단, $M$은 테이블의 크기, $j$는 이동 횟수)

## 3-2. 선형 조사의 치명적 단점: 1차 군집화 (Primary Clustering)
선형 조사는 빈자리를 찾을 때까지 무조건 한 칸씩 이동하기 때문에, 데이터가 특정 영역에 빈틈없이 뭉치는 **1차 군집화(Primary Clustering)** 현상이 필연적으로 발생합니다.

* **원인:** 특정 해시값 주변에 데이터가 쌓이기 시작하면, 다른 해시값을 가진 데이터마저 충돌에 휘말려 그 뭉텅이의 끝에 붙게 됩니다. (눈덩이가 굴러가며 커지는 것과 유사합니다.)
* **결과:** 군집화가 심해지면 비어있는 칸을 찾기 위해 탐색해야 하는 횟수가 기하급수적으로 늘어납니다. 즉, 해시 테이블의 최대 장점인 $O(1)$의 성능이 극단적으로 저하되어 $O(N)$에 가까워지게 됩니다.

## 3-3. 탐색(Search) 시의 작동 원리
데이터를 찾을 때도 데이터를 넣을 때와 똑같은 발자취를 따라갑니다.
1. 해시 함수를 통해 초기 위치(Index)를 찾습니다.
2. 해당 위치에 내가 찾는 키(Key)가 없다면, 다음 칸을 순차적으로 확인합니다.
3. **종료 조건:** 순차 탐색 중 내가 찾는 데이터를 발견하거나, **완전히 비어있는 칸(`null`)을 만나면** 탐색을 종료합니다. (비어있다는 것은 애초에 더 이상 뒤로 밀려나 저장된 데이터가 없다는 뜻이기 때문입니다.)

## 3-4. 자바(Java) 코드 구현 예시

아래는 크기가 $M$인 해시 테이블에서 선형 조사를 구현한 자바 코드입니다.

### ① 전역 변수 및 해시 함수
`0x7fffffff`와 비트 AND 연산을 하여 부호를 없앤 후 테이블 크기 $M$으로 나눈 나머지를 구합니다.

> private int M = 13; // 테이블 크기
> private K[] a = (K[]) new Object[M]; // 키(Key)를 저장할 배열
> private V[] d = (V[]) new Object[M]; // 데이터를 저장할 배열
>
> private int hash(K key){
>     return (key.hashCode() & 0x7fffffff) % M; 
> }

### ② 삽입 연산 (put)
빈 공간(`null`)을 찾거나, 이미 동일한 키가 있으면 값을 갱신하고, 둘 다 아니면 다음 칸으로 이동(`j++`)하는 로직입니다. 

> private void put(K key, V data) {
>     int initialpos = hash(key); // 초기 위치 지정
>     int i = initialpos, j = 1;
>     do {
>         if (a[i] == null) { // 빈자리 발견
>             a[i] = key;
>             d[i] = data;
>             return;
>         }
>         if (a[i].equals(key)) { // 이미 동일한 Key가 존재하면 덮어쓰기
>             d[i] = data;
>             return;
>         }
>         i = (initialpos + j++) % M; // 자리가 없으면 다음 위치로 (원형 탐색)
>     } while (i != initialpos); // 한 바퀴를 다 돌면 종료 (테이블이 꽉 찬 경우)
> }

### ③ 탐색 연산 (get)
초기 위치부터 순차적으로 탐색하며, 빈 칸(`null`)을 만나면 데이터가 없다고 판단하여 `null`을 반환합니다.

> public V get(K key) {
>     int initialpos = hash(key);
>     int i = initialpos, j = 1;
>     while (a[i] != null) { // 빈 칸을 만나기 전까지 반복
>         if (a[i].equals(key)) {
>             return d[i]; // 키를 찾으면 해당 데이터 반환
>         }
>         i = (initialpos + j++) % M; // 다음 위치로 이동
>     }
>     return null; // 탐색 실패 (데이터 없음)
> }

## 4. 개방 주소 방식: 이차 조사 (Quadratic Probing)

## 4-1. 이차 조사(Quadratic Probing)란?
선형 조사의 '1차 군집화(Primary Clustering)' 문제를 해결하기 위해 고안된 방식입니다. "바로 다음 칸에 넣으니까 자꾸 뭉친다면, 이동 폭을 제곱수($1^2, 2^2, 3^2...$)로 확 늘려보자!"라는 아이디어에서 출발했습니다.

* **탐색 수식:** $h_i(x) = (h_0(x) + j^2) \pmod M \quad (j = 0, 1, 2, 3...)$
* 즉, 충돌이 발생하면 초기 위치로부터 1칸, 4칸, 9칸, 16칸... 씩 멀어지며 빈자리를 찾습니다.

## 4-2. 이차 조사의 한계와 문제점
선형 조사의 단점을 보완했지만, 이차 조사 역시 완벽하지 않으며 새로운 문제점들을 야기합니다.

### ① 2차 군집화 (Secondary Clustering)
* 인접한 빈 공간이 채워지며 눈덩이처럼 뭉치는 1차 군집화는 해결했습니다.
* 하지만, **처음부터 같은 해시값을 배정받은 동의어(Synonym) 데이터들**은 이동하는 점프 시퀀스($+1, +4, +9...$)마저 완전히 똑같습니다. 
* 결국 같은 경로를 따라 이동하며 또 다른 형태의 군집을 형성하게 되는데, 이를 **2차 군집화**라고 합니다.

### ② 빈 공간 탐색 실패 (안 보는 칸 발생)
* 선형 조사는 1칸씩 이동하므로 시간이 걸려도 결국 배열의 모든 칸을 한 번씩 다 훑어봅니다.
* 반면 이차 조사는 점프 크기가 $1, 4, 9, 16...$으로 급격히 커지기 때문에, **배열에 분명히 빈자리가 남아 있음에도 불구하고 그 칸을 계속 건너뛰어 영영 찾지 못하는 치명적인 문제**가 발생할 수 있습니다.

### ③ 연산 오버헤드 (속도 저하)
* 선형 조사의 단순 덧셈과 달리, 매 탐색마다 제곱값(곱셈)을 계산해야 하므로 연산 시간이 추가되어 미세한 성능 저하가 발생합니다.


## 4-3. 자바(Java) 코드 구현 예시
아래는 이차 조사를 적용한 해시 테이블의 자바 구현 코드입니다. 무한 루프(빈 공간을 영영 찾지 못하는 상황)를 방지하기 위해 데이터의 수(`N`)가 테이블 크기(`M`)보다 작을 때만 반복하도록 처리합니다.

### ① 삽입 연산 (put)

> private void put(K key, V data) {
>     int initialpos = hash(key); // 초기 위치 지정
>     int i = initialpos, j = 1;
>     do {
>         if (a[i] == null) { // 삽입 위치 발견
>             a[i] = key;
>             d[i] = data;
>             N++; // 항목 수 증가
>             return;
>         }
>         if (a[i].equals(key)) { // 이미 동일한 Key가 존재하면 데이터만 갱신
>             d[i] = data;
>             return;
>         }
>         // 다음 위치 계산: 초기 위치 + (j의 제곱)
>         i = (initialpos + j * j++) % M; 
>     } while (N < M); // 테이블이 가득 차지 않은 경우에만 반복
> }

### ② 탐색 연산 (get)

> public V get(K key) {
>     int initialpos = hash(key);
>     int i = initialpos, j = 1;
>     while (a[i] != null) { // 빈 칸(empty)이 아니면 계속 탐색
>         if (a[i].equals(key)) {
>             return d[i]; // 탐색 성공
>         }
>         // 다음 위치 계산: 초기 위치 + (j의 제곱)
>         i = (initialpos + j * j++) % M; 
>     }
>     return null; // 탐색 실패 (데이터 없음)
> }

## 5. 개방 주소 방식: 이중 해싱 (Double Hashing)

## 5-1. 이중 해싱(Double Hashing)이란?
충돌이 발생했을 때, **해시 함수를 하나 더 도입하여 데이터마다 다른 이동 폭(Probing sequence)을 갖게 하는 방식**입니다. 
선형 조사나 이차 조사가 가진 '군집화(Clustering)' 문제를 근본적으로 해결하기 위해 고안되었습니다.

* **수식:** $h_i(key) = (h_0(key) + i \cdot h'_0(key)) \pmod M \quad (i = 0, 1, 2, \dots)$
    * $h_0(key)$: 기본 해시 함수 (초기 인덱스 결정)
    * $h'_0(key)$: 제2의 해시 함수 (충돌 시 점프할 보폭 결정)

## 5-2. 이중 해싱의 장단점 및 특징

### ① 군집화(Clustering) 완벽 해결
* 같은 초기 해시값을 갖는 동의어(Synonym) 데이터들이라도, **제2의 해시 함수를 거치면 각자 고유한 점프 크기를 갖게 됩니다.** * 따라서 똑같은 시퀀스로 뭉치는 2차 군집화 문제까지 완벽하게 방지할 수 있습니다.

### ② 필수 제약 조건
* **0 반환 금지:** 제2의 해시 함수 $h'_0(key)$는 점프 크기를 정하므로 절대로 0을 리턴해선 안 됩니다. (점프 폭이 0이면 무한 루프에 빠집니다.)
* **서로소(Relatively Prime) 관계:** 해시 테이블의 크기 $M$과 제2의 해시 함수의 결과값은 서로소여야 합니다. 그래야 배열의 특정 칸을 건너뛰지 않고 모든 칸을 골고루 탐색할 수 있습니다. (일반적으로 테이블 크기 $M$을 소수(Prime Number)로 설정하여 이 조건을 만족시킵니다.)

### ③ 단점: 연산 속도 저하 (오버헤드)
* 충돌이 발생할 때마다 두 개의 독립적인 해시 함수를 모두 계산해야 하므로, 단순 덧셈(선형)이나 1번의 곱셈(이차)을 사용하는 **다른 탐색 방식들보다 연산량이 많아 실행 속도가 가장 느리다**는 치명적인 단점이 있습니다.


## 5-3. 구체적인 작동 예시
* **테이블 크기 ($M$):** 13 (소수)
* **제1 해시함수:** $h_1(key) = key \pmod{13}$
* **제2 해시함수:** $d(key) = 7 - (key \pmod 7)$ 
* $key \pmod 7$ 의 결과는 0에서 6 이므로, 이를 7에서 빼면 무조건 1에서 7 사이의 값이 나와 절대 0이 되지 않습니다.

만약 `key`가 35인데 충돌이 났다면?
1. 초기 위치: $35 \pmod{13} = 9$ (이미 데이터가 있다고 가정)
2. 점프 크기: $7 - (35 \pmod 7) = 7 - 0 = 7$
3. 다음 위치: $(9 + 1 \cdot 7) \pmod{13} = 3$ 번 인덱스로 즉시 이동!


## 5-4. 자바(Java) 코드 구현 예시

### ① 삽입 연산 (put)

> private void put(K key, V data) {
>     int initialpos = hash(key);   // 초기 위치
>     int i = initialpos;
>     int j = 1;
>     int d = (7 - (int)key % 7);   // 두 번째 해시 함수 (점프 크기)
>     
>     do {
>         if (a[i] == null) {       // 빈 자리 발견
>             a[i] = key;
>             dt[i] = data;
>             N++;                  // 항목 수 증가
>             return;
>         }
>         if (a[i].equals(key)) {   // 이미 존재하는 키면 데이터만 갱신
>             dt[i] = data;
>             return;
>         }
>         // 다음 위치 계산: 초기 위치 + (j * 점프크기 d)
>         i = (initialpos + j * d) % M; 
>         j++;
>     } while (N < M); // 테이블이 꽉 차기 전까지만 반복
> }

### ② 탐색 연산 (get)

> public V get(K key) {
>     int initialpos = hash(key);
>     int i = initialpos;
>     int j = 1;
>     int d = (7 - (int)key % 7);   // 두 번째 해시 함수 (점프 크기)
>     
>     while (a[i] != null) {
>         if (a[i].equals(key)) {
>             return dt[i];         // 탐색 성공 (데이터 반환)
>         }
>         // 다음 위치 계산: 초기 위치 + (j * 점프크기 d)
>         i = (initialpos + j * d) % M; 
>         j++;
>     }
>     return null;                  // 빈 칸을 만나면 탐색 실패 (null 반환)
> }

## 6. 개방 주소 방식: 랜덤 조사 (Random Probing)

## 6-1. 랜덤 조사(Random Probing)란?
선형 조사($+1, +2, +3...$)나 이차 조사($+1^2, +2^2, +3^2...$)와 같은 규칙적인 점프 시퀀스를 버리고, **의사 난수 생성기(Pseudo-Random Number Generator)** 를 사용하여 무작위한 보폭으로 점프하며 빈자리를 찾는 충돌 해결 방식입니다.

## 6-2. 핵심 원리: "동일한 시드(Seed)와 난수열"
* **탐색의 재현성:** 아무리 무작위로 점프해서 데이터를 저장했더라도, 나중에 그 데이터를 찾으려면 저장할 때 이동했던 경로를 똑같이 따라가야 합니다.
* 즉, **삽입할 때와 탐색할 때 완전히 동일한 랜덤 시퀀스(난수열)가 발생하기만 하면 됩니다.**
* 이를 위해 난수 생성기를 초기화할 때 **항상 동일한 시드(Seed) 값**을 부여하여 예측 가능한 난수를 만들어냅니다.

## 6-3. 랜덤 조사의 한계: 3차 군집화 (Tertiary Clustering)
* 랜덤 조사는 1차 군집화(선형)는 피할 수 있지만, 이차 조사와 유사한 문제점을 공유합니다.
* 초기 해시값이 우연히 같은 동의어(Synonym) 데이터들은 동일한 시드(Seed) 기반의 동일한 난수 시퀀스를 부여받게 됩니다.
* 결국 같은 해시값을 가진 데이터들끼리는 **완전히 똑같은 무작위 점프 경로를 따라 이동하며 또다시 뭉치게 되는데**, 이를 **3차 군집화**라고 부릅니다.


## 6-4. 자바(Java) 코드 구현 예시
삽입(`put`)과 탐색(`get`) 연산 모두에서 `rand.setSeed(10)`으로 동일한 시드를 설정하는 것이 코드의 핵심입니다.

### ① 삽입 연산 (put)

> private void put(K key, V data) {
>     int initialpos = hash(key);   // 초기 위치
>     int i = initialpos;
>     
>     Random rand = new Random();
>     rand.setSeed(10);             // 고정된 시드값 설정 (핵심)
>     
>     do {
>         if (a[i] == null) {       // 빈 자리 발견
>             a[i] = key;
>             d[i] = data;
>             N++;
>             return;
>         }
>         if (a[i].equals(key)) {   // 이미 존재하는 키면 데이터 갱신
>             d[i] = data;
>             return;
>         }
>         // 다음 위치 계산: 초기 위치 + (0~999 사이의 난수)
>         i = (initialpos + rand.nextInt(1000)) % M; 
>     } while (N < M);
> }

### ② 탐색 연산 (get)

> public V get(K key) {
>     Random rand = new Random();
>     rand.setSeed(10);             // 삽입 때와 완벽히 같은 시드값 사용
>     
>     int initialpos = hash(key);   // 초기 위치
>     int i = initialpos;
>     
>     while (a[i] != null) {
>         if (a[i].equals(key)) {
>             return d[i];          // 탐색 성공
>         }
>         // 다음 위치 계산: 삽입 때와 동일한 난수 시퀀스가 발생함
>         i = (initialpos + rand.nextInt(1000)) % M; 
>     }
>     return null;                  // 탐색 실패
> }
