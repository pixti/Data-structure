## 2. 데크 (Deque, Double-ended Queue)
데크는 양쪽 끝(Front, Rear)에서 삽입과 삭제를 모두 허용하는 매우 유연한 선형 자료구조입니다.   
스택(Stack)과 큐(Queue)의 성질을 모두 가지고 있는 혼합 형태입니다.  

### 2-1. 주요 연산 (Deque ADT)
데크는 양방향이기 때문에 각 끝단에 대해 두 종류의 연산이 존재합니다.

| 연산 | 설명 | 관련 자료구조 연산 |
| :--- | :--- | :--- |
| **addFront() / push()** | 데크의 맨 앞(Front)에 원소를 추가 | 스택의 `push` | 
| **addRear() / enqueue()** | 데크의 맨 뒤(Rear)에 원소를 추가 | 큐의 `enqueue` |
| **deleteFront() / dequeue()** | 데크의 맨 앞 원소를 삭제하고 반환 | 큐의 `dequeue` |
| **deleteRear() / pop()** | 데크의 맨 뒤 원소를 삭제하고 반환 | 스택의 `pop` |
| **getFront() / getRear()** | 삭제 없이 맨 앞 또는 맨 뒤의 원소를 확인 | `peek` |

### 2-2. 데크의 응용 (Applications)
양방향에서 데이터 처리가 필요한 다양한 상황에서 사용됩니다.
1. 스크롤(Scroll): 화면을 위아래로 움직일 때 새로운 데이터가 추가되고 오래된 데이터가 나가는 구조.
2. 웹 브라우저 방문 기록: - 새로운 페이지 방문 시 앞에 추가.
 - 기록이 너무 많아지면 가장 오래된 기록을 뒤에서 삭제.
3. 문서 편집기의 Undo (되돌리기): 개수 제한이 있는 실행 취소 목록에서 최근 작업은 추가하고,  
   제한을 넘긴 오래된 작업은 제거할 때 사용.

### 2-3. 구현: 이중 연결 리스트 (Doubly Linked List)
데크를 구현할 때 가장 효율적인 방식은 **이중 연결 리스트(Doubly Linked List)** 를 사용하는 것입니다.  
단순히 "양방향이니까 양방향 리스트를 쓴다"는 것 이상의 아주 중요한 성능상 이유가 있습니다.

### ① 단순 연결 리스트(Singly Linked List)의 치명적 단점: `removeLast()`

단순 연결 리스트로 데크를 구현하면 대부분의 연산은 효율적이지만, **맨 뒤의 원소를 삭제할 때** 성능 저하가 발생합니다.

* **상황:** `rear`가 가리키는 마지막 노드를 삭제해야 함.
* **문제점:** 삭제 후에는 `rear` 포인터를 **'마지막 노드의 바로 앞 노드'** 로 옮겨야 합니다.
* **이유:** 단순 연결 리스트는 이전 노드로 돌아가는 링크(`prev`)가 없습니다.  
  결국 `front`부터 시작해서 `next`를 타고 끝까지 다시 탐색해야 합니다.
* **결과:** 이 탐색 과정 때문에 `removeLast()` 연산은 $O(N)$ 의 시간 복잡도를 가지게 됩니다.  

### ② 이중 연결 리스트(Doubly Linked List)의 해결책

각 노드가 이전 노드를 가리키는 `prev` 참조를 가지고 있으면 상황이 완전히 달라집니다.
* **해결:** `rear = rear.prev` 명령 한 줄이면 즉시 이전 노드로 이동할 수 있습니다.
* **결과:** 모든 연산(삽입/삭제/조회)을 예외 없이 **$O(1)$** 의 시간 복잡도로 수행할 수 있습니다.


| 구분 | 단순 연결 리스트 (Singly) | 이중 연결 리스트 (Doubly) |
| :--- | :---: | :---: |
| **addFront / addLast** | $O(1)$ | $O(1)$ |
| **removeFront** | $O(1)$ | $O(1)$ |
| **removeLast** | **$O(N)$** (치명적 단점) | **$O(1)$** |


## 3. 스택·큐·데크 연산 및 수행시간 비교
선형 자료구조의 핵심인 세 구조는 모두 삽입과 삭제에 최적화되어 있습니다.   
하지만 어느 쪽 끝을 사용할 수 있느냐에 따라 메서드 명칭과 활용도가 달라집니다.  

### 3-1. 주요 메서드 명칭 (Java 기준)

| 기능 | 스택 (Stack) | 큐 (Queue) | 데크 (Deque) |
| :--- | :--- | :--- | :--- |
| **삽입 (앞)** | - | - | `addFirst()` / `push()` |
| **삽입 (뒤)** | `push()` | `enqueue()` / `add()` | `addLast()` / `offer()` |
| **삭제 (앞)** | - | `dequeue()` / `poll()` | `removeFirst()` / `pop()` |
| **삭제 (뒤)** | `pop()` | - | `removeLast()` |
| **조회 (앞)** | - | `peek()` / `front()` | `peekFirst()` / `first()` |
| **조회 (뒤)** | `peek()` / `top()` | - | `peekLast()` / `last()` |
| **상태 확인** | `isEmpty()`, `size()` | `isEmpty()`, `size()` | `isEmpty()`, `size()` |

### 3-2. 연산 수행시간 (Time Complexity)  

모든 연산은 구현 방식에 따라 미세한 차이가 있지만, 기본적으로 **$O(1)$**을 목표로 합니다.

| 연산 구분 | 배열(Array) 기반 | 연결리스트(Linked List) 기반 |
| :--- | :--- | :--- |
| **삽입 (Insertion)** | $O(1)$ * | $O(1)$ |
| **삭제 (Deletion)** | $O(1)$ * | $O(1)$ |
| **조회 (Peek/Front)** | $O(1)$ | $O(1)$ |
| **탐색 (Search)** | **$O(N)$** | **$O(N)$** |
| **비고** | * 리사이징 발생 시 $O(N)$ (상각 $O(1)$ ) | 데크는 반드시 **이중 연결리스트** 필요 | 


### 3-3. 수행시간 분석

#### 1. 삽입과 삭제 ( $O(1)$ )
* **배열:** 인덱스를 알고 있으므로 즉시 접근하여 처리합니다.  
  다만, 배열이 꽉 차서 크기를 늘려야 할 때는 모든 원소를 복사해야 하므로 일시적으로 $O(N)$ 이 걸립니다.  
 이를 평균적으로 계산하면 상각 분석(Amortized Analysis)에 의해 $O(1)$이 됩니다.  
* **연결리스트:** 포인터(레퍼런스)만 변경하면 되므로 언제나 일정한 $O(1)$을 보장합니다.  

#### 2. 조회 (Peek / Top / Front / First / Last)
* 자료구조의 '입구' 혹은 '출구'에 있는 데이터를 확인만 하는 것이므로,   
 포인터나 인덱스를 통해 즉시 접근하여 $O(1)$에 끝납니다.   

#### 3. 탐색 (Search) 및 임의 접근 ($O(N)$)
* **주의:** 스택, 큐, 데크는 "가장 앞"이나 "가장 뒤"가 아닌 **중간에 있는 데이터를 찾는 용도가 아닙니다.**   
특정 값을 찾으려면 결국 모든 데이터를 하나씩 다 꺼내보거나 훑어야 하므로 $O(N)$의 시간이 소요됩니다.  
(이런 용도라면 리스트나 트리, 해시 테이블을 써야 합니다.)
