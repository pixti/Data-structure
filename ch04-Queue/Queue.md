# 1. 큐 (Queue)
큐(Queue)는 **FIFO (First-In First-Out, 선입선출)** 이라는 원칙을 가진 선형 자료구조입니다. 가장 먼저 들어온 데이터가 가장 먼저 나가는 '대기열' 형태를 가집니다.

## 1-1. 주요 연산 (Queue ADT)
큐도 스택과 마찬가지로 추상 자료형(ADT)입니다. 구현 방식에 상관없이 아래의 공통적인 명세를 따릅니다.

| 연산 | 설명 |
| :--- | :--- |
| **enqueue(item)** | 큐의 맨 끝(Rear)에 새로운 원소를 추가함 |
| **dequeue()** | 큐의 맨 앞(Front)의 원소를 삭제하고 그 값을 반환함 |
| **peek() / front()** | 맨 앞의 원소를 삭제하지 않고 값만 읽어옴 |
| **isEmpty()** | 큐가 비어있는지 확인 (비었으면 true) |
| **dequeueAll()** | 큐의 모든 항목을 한 번에 제거함 |

> **주의**: 큐 또한 스택처럼 **검색(Search)** 기능을 설계하지 않습니다. 오직 데이터의 순차적인 흐름에만 집중하기 때문입니다.

## 1-2. 큐의 핵심 포인터: front와 rear
배열로 큐를 구현할 때, 데이터의 위치를 추적하기 위해 두 개의 포인터가 필요합니다.

* **front (앞)**: 큐에서 데이터를 꺼낼 위치(출구)를 가리킵니다.
   (표준 : 첫 번째 데이터의 **바로 앞** 빈 공간을 가리킵니다.)
* **rear (뒤)**: 큐에 데이터를 넣을 위치(입구)를 가리킵니다.
   (표준 : 마지막으로 삽입된 데이터의 인덱스를 가리킵니다.)

### 데이터가 들어오고 나갈 때의 움직임
1. **초기 상태**: `front`와 `rear`는 보통 `-1` 또는 `0`에서 시작합니다. (둘의 값이 같으면 비어있다는 뜻)
2. **데이터 삽입 (Enqueue)**: 새로운 데이터가 들어오면 `rear`를 한 칸 뒤로 밀고 그 자리에 데이터를 씁니다. (`rear++`)
3. **데이터 삭제 (Dequeue)**: 데이터를 내보내면 `front`를 한 칸 뒤로 밀어서 다음 데이터를 출구로 지정합니다. (`front++`)


## 1-3. 선형 큐(Linear Queue)의 한계와 문제점
단순한 배열로 큐를 구현하면 `front`와 `rear`가 뒤쪽 방향으로만 전진합니다.

### ① 가짜 포화 상태 (Memory Drift)
데이터를 넣고 빼다 보면 `front`와 `rear`는 계속 배열의 오른쪽(큰 인덱스)으로 이동합니다.
* **상황**: 배열 크기가 5일 때, 5번 넣고 3번 뺐다면?
* **결과**: `rear`는 이미 인덱스 4(끝)에 도달했습니다. 배열 앞쪽(0, 1, 2)은 비어있지만,  
* `rear`가 끝이라는 이유로 **"큐가 꽉 찼다"** 고 판단합니다.  
* **결론**: 공간이 남아도 활용하지 못하는 **메모리 낭비**가 발생합니다.  

### ② [해결 방안 1] 데이터 이동 (Shifting) - "비효율의 극치"  
`rear`가 끝에 도달했을 때, 모든 데이터를 0번 인덱스 방향으로 당기는 방식입니다.  
* **문제점**: 데이터가 $N$ 개일 때 삭제마다 $O(N)$ 의 시간이 걸려 큐의 성능 장점이 사라집니다.  

## 1-4. 원형 큐 (Circular Queue)  
**[해결 방안 2]**  
`rear`가 배열의 끝(`capacity-1`)에 도달했을 때, 다음 삽입 위치를 다시 0번 인덱스로 돌려보내 배열을 원형처럼 사용합니다.  

### ① 인덱스 순환: 나머지 연산(`%`)  
* **`nextIndex = (i + 1) % N`** (N은 배열의 크기)  
* 예: `N=8`일 때 인덱스 `7` 다음은 `(7 + 1) % 8 = 0`이 됩니다. 

### ② 설계 이슈: Empty vs Full 구분  
원형 큐에서 `front == rear`는 두 가지 상반된 상태를 의미할 수 있습니다.

 상태의 모호성 :
    **Empty** : 데이터를 다 지워서 `front`가 `rear`를 따라잡았을 때.
    **Full** : 데이터를 다 채워서 `rear`가 한 바퀴 돌아 `front`를 만났을 때.  
    
큐가 **비어있음(Empty)** 에도 불구하고, **`rear`는 마지막으로 삭제된 item의 위치를 여전히 가리키고 있습니다.**  
   즉, 인덱스상으로는 어딘가를 가리키고 있지만 논리적으로는 비어있는 상태가 되어 `Full` 상태와 구분이 불가능해집니다.

* **[방법 1] 직전 연산 기억**: 마지막 연산이 `enqueue`였고 포인터가 같아지면 Full, `dequeue`였으면 Empty.  
* **[방법 2] 카운트 변수 활용**: `size` 변수를 두어 데이터 개수를 직접 체크.  
* **[방법 3] 한 칸을 비워두기 (표준)**:  
    * `front`를 항상 **첫 번째 데이터의 바로 앞 칸**에 둡니다.  
    * **Empty**: `front == rear`  
    * **Full**: `(rear + 1) % capacity == front`  
    * 한 칸을 희생하지만 로직이 가장 빠르고 명확합니다.  

## 1-5. 큐의 Resize의 문제점 
원형 큐에서 배열의 크기를 키울 때 가장 큰 문제는 **"논리적인 순서와 물리적인 위치가 어긋나 있다"** 는 점입니다. 
이를 해결하기 위해 'Unwrapping(펼치기)' 과정이 필요합니다.

### 1. 왜 단순히 배열만 키우면 안되는가? (문제점)  
원형 큐는 `rear`가 배열의 끝에 도달하면 다시 0번 인덱스로 돌아옵니다. 여기서 문제가 발생합니다.  

* **예시 상황**: 기존 배열(크기 8)에서 데이터가 `[ C, D, E, F, G, - , A, B ]` 순으로 들어있습니다. (front=5, rear=4)  
* **단순 확장 시**: 배열 크기만 16으로 늘리면, `A, B`는 인덱스 6, 7에 있고 `C, D, E, F, G`는 인덱스 0~4에 있게 됩니다.  
  ` [ C, D, E, F, G, - , A, B, -, -, -, -, -, -, -, - ]  `
* **결과**: 큐의 논리적 순서는 `A → B → C → D → E → F → G`인데, 단순 복사 후 다음 데이터를 넣으려고 하면  
  인덱스 5(기존 front 자리)부터 채워지게 되어  **데이터의 선입선출(FIFO) 순서가 완전히 뒤엉키게 됩니다.**    

### 2. 해결 방법: 데이터 재배치 (Unwrapping) 로직 
논리적 순서를 유지하기 위해, 새 배열에는 데이터를 **인덱스 0번부터 일렬로** 새로 쌓아야 합니다.  

#### ① 새 배열 생성
* 기존 배열 크기의 2배(예: 8 → 16)인 `newQueue`를 생성합니다. 

#### ② Part 1 복사 (기존 배열의 후반부)  
* **대상**: 기존 배열에서 `front + 1` 인덱스부터 배열의 끝(`capacity - 1`)까지.  
* **이유**: 원형 큐에서 이 구간은 **현재 큐에서 가장 오래된(먼저 나갈) 데이터들**입니다.
* **복사 위치**: 새 배열의 **0번 인덱스**부터 차례대로 넣습니다.  
  `[ C, D, E, F, G, - , A, B ]` -> ` [ A, B, - , - , - , - , - , - , -, -, -, -, -, -, -, - ]  `  
* 인덱스 6, 7에 있던 `A, B`를 새 배열의 `[0], [1]`번으로 복사.

#### ③ Part 2 복사 (기존 배열의 전반부)
* **대상**: 기존 배열의 `0`번 인덱스부터 `rear`까지.  
* **이유**: 배열 끝을 돌아(Wrap-around) 뒤늦게 들어온 데이터들입니다.  
* **복사 위치**: 새 배열에서 **Part 1 복사가 끝난 지점 바로 뒤**에 이어서 붙입니다.
  `[ C, D, E, F, G, - , A, B ]` -> ` [ A, B, C, D, E, F, G, - , -, -, -, -, -, -, -, - ]  `   
* 인덱스 0 ~ 4에 있던 `C, D, E, F, G` 를 새 배열의 `[2]~[6]`번으로 복사.  

#### ④ 포인터 갱신 (Pointers Reset)
* 이제 모든 데이터가 새 배열의 앞부분에 순서대로(A, B, C, D, E, F, G) 정렬되었습니다.  
* **front 갱신**: 데이터를 0번부터 채웠으므로,  
 `front`는 다시 **첫 데이터의 바로 앞 칸인 0번의 이전 혹은 배열의 마지막 인덱스**로 설정합니다.    
 (가장 깔끔하게는 `front = 새 배열 크기 - 1` 혹은 데이터가 0부터 시작하도록 `front = 0`의 앞인 마지막 인덱스로 보냅니다.)
* **rear 갱신**: 데이터 개수만큼 채워졌으므로 `rear = (전체 데이터 개수 - 1)`이 됩니다.
