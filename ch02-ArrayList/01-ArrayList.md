## 1. 배열(Array)

## 1-1. 배열(Array)이란?
배열(Array)은 동일한 타입의 데이터들을 메모리상의 **연속적인 공간**에 나열하여 저장하는 가장 기본적인 **선형(Linear) 자료구조**입니다.

### 핵심 개념
* **인덱스(Index):** 각 원소는 고유한 숫자인 인덱스로 식별됩니다.
* **연속적 할당:** 데이터가 메모리에 순차적으로 저장되므로, **논리적 순서(Index)와 물리적 순서(Memory Address)가 일치**합니다.
* **고정 선언:** 배열을 선언할 때 요소의 타입(Type)과 길이(Length)를 반드시 지정해야 합니다.

## 1-2. 배열(Array)의 장점
### ① 임의 접근(Random Access)을 통한 빠른 탐색
* 인덱스를 사용하여 특정 위치의 원소에 즉시 접근할 수 있습니다.
* **시간 복잡도:** $O(1)$

### ② 빠른 데이터 수정
* 수정하고자 하는 위치의 인덱스를 알고 있다면, 그 index에 새로운 데이터를 덮어쓰는 작업만 수행하면 됩니다.
* **시간 복잡도:** $O(1)$

## 1-3. 배열(Array)의 단점

### ① 정적 크기(Static Size)
* 선언 시 지정한 length는 고정된 값입니다. 한 번 정해진 length는 실행 도중에 늘리거나 줄일 수 없습니다.
### ② 타입 제한
* 선언 시 지정한 데이터 type만 저장할 수 있으며, 다른 type의 데이터를 혼합하여 저장할 수 없습니다.

### ③ 삽입 및 삭제의 비효율성
* **중간 삽입/삭제:** 연속적인 구조를 유지해야 하므로, 특정 위치에 데이터를 넣거나 뺄 때 해당 index부터 그 뒤의 데이터들을 모두 뒤로 밀거나 앞으로 당겨야 합니다.
* **시간 복잡도:** $O(N)$
* 단, 배열의 맨 뒤에 데이터를 삽입하거나 삭제하는 것은 이동이 필요 없으므로 $O(1)$이 소요됩니다.

## 1-4. 배열(Array) 시간 복잡도 요약

| 연산 유형 | 시간 복잡도 | 설명 |
| :--- | :--- | :--- |
| **접근 (Access)** | $O(1)$ | 인덱스를 통한 직접 접근 (Random Access) |
| **수정 (Update)** | $O(1)$ | 특정 인덱스의 값 변경 |
| **삽입 (Insert)** | $O(N)$ | 데이터 Shifting 필요 (맨 뒤 삽입은 O(1)) |
| **삭제 (Delete)** | $O(N)$ | 데이터 Shifting 필요 (맨 뒤 삭제는 O(1)) |
| **검색 (Search)** | $O(N)$ | 인덱스를 모를 경우 순차적으로 찾아야 함 |

## 2. 기본 자료형 배열 (Primitive Array) vs 참조 자료형 배열 (Reference Array)

배열은 어떤 데이터를 담느냐에 따라 크게 **기본 자료형 배열**과 **참조 자료형 배열**로 나뉩니다.  
두 방식은 메모리에 데이터가 저장되는 형태에서 큰 차이를 보입니다.  

## 2-1. 기본 자료형 배열 (Primitive Array)
기본 자료형(Primitive Type)의 **실제 값**을 배열의 각 칸에 직접 저장합니다.

* **종류:** `byte`, `short`, `int`, `long`, `float`, `double`, `char`, `boolean` 등
* **특징:** 메모리 공간에 실제 데이터 값이 들어있으므로 접근 속도가 매우 빠르고 메모리 효율적입니다.

## 2-2. 참조 자료형 배열 (Reference Array)
객체(Object)에 대한 메모리 주소(Reference)를 배열의 각 칸에 저장합니다.

* **저장 대상:**
    * **사용자 정의 클래스:** 예) `Student`, `Car` 등
    * **래퍼 클래스(Wrapper Class):** 기본 자료형을 객체로 다루기 위해 사용하는 클래스 (`Integer`, `Double`, `Character`, `Boolean` 등)
* **작동 방식:** 실제 객체 데이터는 메모리의 다른 영역(Heap 등)에 저장되고, 배열 칸에는 그 데이터가 위치한 '주소값'만 담깁니다.

## 2-3. 왜 참조 배열은 주소값을 담을까?
배열의 가장 큰 장점은 임의 접근(Random Access)입니다. 특정 인덱스로 즉시 이동하기 위해서는 **모든 칸의 크기가 일정**해야 합니다.  
* **문제점:** 객체(Object)는 내부에 담긴 필드나 메서드에 따라 크기가 제각각입니다.  
 만약 배열 칸에 실제 객체를 통째로 넣으려 한다면 칸마다 크기가 달라져 인덱스로 위치를 계산할 수 없게 됩니다.  
* **해결책:** 실제 데이터의 크기가 다르더라도, 그 데이터를 가리키는 **주소(Reference)의 크기는 모두 동일**합니다.  
  따라서 배열에는 일정한 크기의 '주소값'만 저장하여 $O(1)$의 탐색 성능을 유지합니다.

## 2-4. 기본 자료형 배열 (Primitive Array) vs 참조 자료형 배열 (Reference Array) 요약

| 구분 | 기본 자료형 배열 | 참조 자료형 배열 |
| :--- | :--- | :--- |
| **저장 내용** | 실제 데이터 값 (Value) | 객체의 주소값 (Reference) |
| **칸의 크기** | 자료형의 크기 (ex. int는 4byte) | 주소값의 크기 (환경에 따라 고정) |
| **실제 데이터 위치** | 배열 공간 내부 | 메모리의 별도 영역 (Heap 등) |
| **예시** | `int[] arr = {1, 2, 3};` | `Student[] arr = new Student[3];` |

## 3. 배열의 인덱스와 메모리 주소 계산 원리

배열이 왜 탐색에 강점을 가지는지, 컴퓨터가 내부적으로 인덱스를 어떻게 주소로 변환하는지 그 수학적 원리를 정리합니다.

## 3-1. 메모리 주소 계산 공식
배열의 특정 인덱스(`index`)에 접근할 때, 컴퓨터는 시작 주소로부터 해당 위치까지의 거리를 단 한 번의 산술 연산으로 계산합니다.

$$Address(i) = Base + (Size \times i)$$

* **$Base$ (a):** 배열의 시작 주소 (0번 인덱스의 주소)
* **$Size$:** 배열에 저장된 데이터 타입의 크기 (byte 단위)
* **$i$ (index):** 접근하고자 하는 요소의 번호


## 3-2. 구체적인 예시: `int`형 배열
정수형(`int`) 배열을 예로 들어 계산 과정을 살펴보겠습니다.

### 가정 환경
* **배열의 시작 주소 ($Base$):** `100`번지
* **데이터 타입 크기 ($Size$):** `int`형은 한 칸에 **4byte** 차지
* **찾고자 하는 위치:** `3번 인덱스`

### 계산 과정
컴퓨터는 1번지, 2번지를 차례대로 거치지 않고 아래 식을 통해 목표 주소로 즉시 점프(Jump)합니다.

$$100 + (4 \times 3) = 112$$

따라서 컴퓨터는 메모리의 **112번지**로 바로 접근하여 데이터를 읽어옵니다.


## 3-3. 데이터 타입별 점프 크기
배열을 선언할 때 타입을 지정해야 하는 이유는, 타입에 따라 메모리 한 칸의 **Size**가 달라지기 때문입니다.

| 데이터 타입 | 크기 (Byte) | 주소 계산 단위 |
| :--- | :--- | :--- |
| **char** | 2 byte | $Base + (2 \times i)$ |
| **int** | 4 byte | $Base + (4 \times i)$ |
| **long** | 8 byte | $Base + (8 \times i)$ |
| **Reference** | 4 or 8 byte | 주소값 자체의 크기만큼 점프 |


## 3-4. 왜 배열의 탐색 성능은 항상 $O(1)$인가?

배열의 데이터 개수($n$)가 10개든, 1억 개든 상관없이 주소를 찾는 과정은 항상 [곱셈 1번 + 덧셈 1번]이라는 단순한 산술 연산으로 끝납니다.

1.  **상수 시간 소요:** 데이터 양이 늘어나도 연산 횟수가 변하지 않습니다.
2.  **직접 접근:** 인덱스를 주소로 바로 변환하여 해당 위치로 즉시 이동하는 **임의 접근(Random Access)** 방식이기 때문입니다.

결론적으로, 배열은 메모리상에 **연속적**으로 데이터를 배치함으로써 물리적인 주소 계산을 최적화한 자료구조입니다.

## 4. 리스트 ADT(Abstract Data Type)란?

자료구조를 공부할 때 가장 먼저 접하는 개념 중 하나인 ADT(추상 자료형)와 그 대표적인 예시인 **List ADT**에 대해 정리합니다.


## 4-1. ADT(Abstract Data Type, 추상 자료형)란?

ADT는 데이터의 형태와 그 데이터에 대해 수행할 수 있는 연산들을 **수학적으로 명세**한 것입니다. 가장 중요한 핵심은 **'구현(Implementation)'과 '명세(Specification)'를 분리**하는 것입니다.

* **무엇을(What):** 어떤 데이터를 담고, 어떤 작업을 수행할 수 있는지 정의합니다. (설명서 역할)
* **어떻게(How):** 내부적으로 어떻게 코드로 구현되었는지는 고려하지 않습니다. (구현 상세는 은닉)

> ** 비유하자면?**
> 우리가 세탁기를 사용할 때 '전원을 켜고, 모드를 선택하고, 동작 버튼을 누른다'는 기능(ADT)만 알면 되지, 세탁기 내부의 모터가 어떻게 돌아가고 회로가 어떻게 구성되어 있는지(Implementation)는 알 필요가 없는 것과 같습니다.


## 4-2. 리스트 ADT (List ADT)

리스트 ADT는 일련의 동일한 타입의 아이템들을 위치 순서(Positional Order) 대로 늘어놓은 자료구조를 말합니다.

### 주요 특징
* **선형 구조:** 데이터들이 일렬로 나열되어 있습니다.
* **순서 유지:** 데이터 간에 앞뒤 관계가 명확하며, 각 원소는 위치(Index)를 가집니다.
* **유연성:** 리스트의 크기는 고정되어 있지 않다고 가정하며(추상적 단계), 다양한 연산을 지원합니다.

## 4-3. 리스트 ADT의 주요 기능 (Operations)

리스트가 "무엇을 할 수 있는가"에 대한 명세입니다.

| 기능 | 설명 |
| :--- | :--- |
| **insert(i, x)** | $i$번 자리에 원소 $x$를 삽입합니다. |
| **delete(i)** | $i$번 자리의 원소를 삭제합니다. |
| **remove(x)** | 리스트에서 원소 $x$를 찾아 삭제합니다. |
| **get(i)** | $i$번 자리에 있는 원소가 무엇인지 반환합니다. |
| **indexOf(x)** | 원소 $x$가 리스트의 몇 번 위치에 있는지 확인합니다. |
| **size()** | 현재 리스트에 저장된 원소의 개수를 알려줍니다. |
| **isEmpty()** | 리스트가 비어있는지 확인합니다. |

## 4-4. 왜 ADT가 중요한가요?
1.  **추상화(Abstraction):** 복잡한 내부 구현을 신경 쓰지 않고, 데이터의 구조와 기능에만 집중할 수 있습니다.
2.  **교체 가능성:** 리스트 ADT를 구현하는 방식은 배열(Array)이 될 수도 있고, 연결 리스트(Linked List)가 될 수도 있습니다.  
   사용자는 인터페이스(ADT)만 같다면 내부 구현이 바뀌어도 코드를 수정할 필요가 없습니다.
3.  **설계 우선:** 코드를 짜기 전 자료구조의 청사진을 그리는 단계로 활용됩니다.
>  리스트 ADT는 데이터를 순서대로 관리하기 위한 '기능 설명서'입니다. 이를 실제로 메모리에 어떻게 올릴 것인가(배열 vs 연결 리스트)는 다음 단계인 구현의 영역입니다.

## 5. ArrayList(배열 리스트)

**ArrayList**는 내부적으로 배열(Array)을 사용하여 **List ADT**를 구현한 자료구조입니다.   
기존 배열의 강력한 장점인 인덱스 접근을 유지하면서도, '고정 크기'라는 치명적인 단점을 보완한 형태입니다.

## 5-1. 동적 배열(Dynamic Array)과 리사이징(Resizing)

일반적인 배열은 선언 시 크기가 고정되어 있어 공간이 꽉 차면 **Overflow**가 발생합니다.   
하지만 ArrayList는 공간이 부족할 경우 자동으로 크기를 조절하는 **Resizing** 전략을 사용합니다.

## Resizing 원칙 (통념적 기준)
개발자들 사이에서 공간 효율과 성능의 균형을 맞추기 위해 통용되는 기준은 다음과 같습니다.

* **확장(Expansion):** 배열이 꽉 차면 기존 배열 크기의 **2배**인 새로운 배열을 할당합니다.
* **축소(Reduction):** 배열의 **3/4이 비어 있을 때** (즉, 1/4만 사용 중일 때) 배열 크기를 **1/2로 축소**합니다.
    * *Tip: 1/2이 되자마자 바로 줄이지 않는 이유는, 데이터를 하나 넣고 빼는 작업이 반복될 때 계속해서 확장/축소가 일어나는 **진동(Oscillation) 문제**를 방지하기 위함입니다.*

## 5-2. ArrayList의 내부 동작

List ADT의 `insert(i, x)` 기능이 리스트 크기에 관계없이 동작할 수 있는 이유는 내부적인 **데이터 복사** 과정 덕분입니다.

1.  공간이 부족하면 더 큰 새로운 배열을 생성합니다.
2.  기존 배열의 데이터를 새로운 배열로 모두 복사(O(N))합니다.
3.  기존 배열을 삭제하고 새 배열을 참조합니다.

## 5-3. 연산 및 시간 복잡도 (Time Complexity)

### ① 조회 및 수정: $O(1)$
* 내부가 배열이므로 인덱스를 통한 임의 접근(Random Access)이 가능합니다.

### ② 삽입 및 삭제 (중간): $O(N)$
* 데이터를 삽입하거나 삭제할 때, 연속적인 구조를 유지하기 위해 원소들을 밀고 당기는 **시프팅(Shifting)** 과정이 필요합니다.
* **최적화 팁:** 만약 데이터의 저장 순서가 중요하지 않다면, 삭제된 자리에 맨 뒤의 원소를 옮겨와 채우는 방식으로 $O(1)$에 구현할 수도 있습니다.


### ③ 맨 뒤 삽입(Add Last): Amortized $O(1)$
* 일반적으로는 빈 공간에 넣기만 하면 되므로 $O(1)$입니다.
* 공간이 다 차서 **Resizing**이 발생하는 순간에는 $O(N)$이 소요됩니다.
* 하지만 리사이징은 매우 드물게 발생하므로, 전체 연산 횟수로 나누어 평균을 내면 **분할 상환(Amortized) $O(1)$**의 성능을 보입니다.

## 5-4. 요약 테이블

| 연산 | 시간 복잡도 | 비고 |
| :--- | :--- | :--- |
| **Get / Set (Index)** | $O(1)$ | 인덱스를 통한 직접 접근 |
| **Add / Remove (Middle)** | $O(N)$ | 데이터 시프팅(Shifting) 발생 |
| **Add (End)** | **Amortized $O(1)$** | 대부분 $O(1)$이나 리사이징 시 $O(N)$ |
| **Resizing** | $O(N)$ | 새로운 배열 할당 및 데이터 복사 |

## 5-5. ArrayList의 장단점

### 장점
* 데이터 조회가 매우 빠릅니다 (O(1)).
* 데이터 추가 시 크기를 신경 쓰지 않아도 됩니다.

### 단점
* Array의 단점이 그대로 적용되어, 중간 삽입/삭제 성능이 좋지 않습니다.
* Resizing 과정에서 일시적인 성능 저하와 메모리 복사가 발생합니다.
* 배열 크기를 미리 크게 잡아둘 경우 메모리 낭비가 생길 수 있습니다.

---
> **정리:** ArrayList는 배열의 '빠른 조회'라는 장점과 리스트의 '유연한 크기'라는 장점을 결합한 형태입니다. 조회가 많고 데이터의 추가/삭제가 주로 맨 뒤에서 일어나는 경우 가장 효율적인 자료구조입니다.
