## 10. 스레드 이진 트리 (Threaded Binary Tree)

### 10-1. 등장 배경 및 필요성
이진 트리의 기본 연산들은 레벨 순회를 제외하고 대부분 **스택(Stack)** 자료구조를 사용합니다.   
메소드의 재귀 호출은 시스템 스택을 사용하므로, 스택에 사용되는 메모리 공간의 크기는 **트리의 높이($H$)에 비례**하게 됩니다.  

**[스택 없이 이진 트리 연산을 구현하는 2가지 방법]**  
1.  **부모 노드 레퍼런스 활용**: 각 노드 객체에 부모 노드를 가리키는 필드를 추가하여 순회 시 거슬러 올라가는 방식입니다.  
2.  **스레드 이진 트리**: 노드의 **null 레퍼런스** 공간을 활용하여 순회 경로를 직접 저장하는 방식입니다.  

> **※참고**: 여기서의 '스레드'는 운영체제(OS)에서 실행 흐름을 나누는 스레드(Thread)와는 전혀 무관한 용어입니다.

### 11-2. Null 레퍼런스 필드 수 = N+1  
$N$개의 노드가 있는 이진 트리에서 버려지는 null 레퍼런스 필드의 수는 항상 $N+1$개이며, 이를 활용하는 것이 스레드 트리의 핵심입니다.   

* **총 레퍼런스 필드 수**: $2N$ (각 노드마다 left, right 필드 존재)  
* **부모-자식 연결 레퍼런스 수**: $N-1$  
    * 루트 노드를 제외한 모든 노드는 정확히 1개의 부모 노드를 갖기 때문입니다.  
* **수식**: $2N - (N-1) = \mathbf{N+1}$  

### 11-3. 핵심 개념 및 작동 원리  
**스레드 이진 트리**는 앞서 계산된 $N+1$개의 null 레퍼런스를 활용하여,  
중위 순회 시 **이전에 방문한 노드**와 **다음에 방문할 노드**를 가리키도록 설계된 트리입니다.  

* **스레드(Thread)**: null 레퍼런스 공간에 저장된 선행자/후속자의 주소값입니다.  
* **태그(Tag) 필드**: 레퍼런스가 '실제 자식'인지 '스레드'인지 구분하기 위해 2개의 boolean 필드를 사용합니다.  

| 태그 필드명 | 값 (true / false) | 의미 설명 |
| :--- | :--- | :--- |
| **isLeftThread** | **true (t)** | 왼쪽 레퍼런스가 **중위 선행자**를 가리킴 |
| | **false (f)** | 왼쪽 레퍼런스가 실제 **왼쪽 자식**을 가리킴 |
| **isRightThread** | **true (t)** | 오른쪽 레퍼런스가 **중위 후속자**를 가리킴 |
| | **false (f)** | 오른쪽 레퍼런스가 실제 **오른쪽 자식**을 가리킴 |

### 11-4. 중위 순회 스레드 이진 트리 (Inorder Threaded)  
대부분 중위 순회를 기반으로 구현하며, 데이터 구조상 다음과 같은 화살표 방향을 갖습니다.
 
* **중위 선행자 (Predecessor)**: 중위 순회 시 현재 노드 바로 앞에 방문하는 노드입니다. (이미지의 점선 화살표)   
* **중위 후속자 (Successor)**: 중위 순회 시 현재 노드 바로 다음에 방문하는 노드입니다. (이미지의 실선 화살표)

### 11-5. 장점 및 단점  

* **장점**  
    * 재귀나 스택을 사용하는 순회보다 실행 속도가 빠릅니다.  
    * 추가적인 스택 메모리가 필요 없어 **공간 효율성**이 매우 뛰어납니다.  
* **단점**  
    * 노드 삽입과 삭제가 빈번한 경우, 스레드(포인터)와 태그 필드를 매번 갱신해야 하므로 구현이 복잡합니다.  
    * 트리 구조 변경 시 연산 오버헤드가 커서 동적 데이터 환경에서는 성능이 저하될 수 있습니다.  

### 11-6. 복잡도 분석

* **시간 복잡도**: $O(N)$
    * 트리의 모든 노드를 방문해야 하므로 노드 수 $N$에 비례합니다. 하지만 스택 관리 비용이 없어 상수 시간 성능이 개선됩니다.
* **공간 복잡도**: **$O(1)$**
    * 일반 순회는 재귀 호출 스택으로 인해 $O(H)$의 공간을 사용하지만, 스레드 트리는 이미 존재하는 포인터를 따라가기만 하므로 추가 공간이 거의 들지 않습니다.
 

## 12. 승자 트리 (Winner Tree)
### 12-1. 기본 개념 및 용어 정의
승자 트리는 $k$개의 외부적으로 정렬된 데이터 **런(run)** 을 병합(merge)할 때 가장 작은 키 값을 가진 노드를 결정하는 데 사용되는 **완전 이진 트리**입니다.  

* **런(run)**: 하나의 순서 순차로 합병될 $k$개의 순서 순차입니다. 각 런은 키 필드에 따라 **비감소 순서(오름차순)** 로 이미 정렬되어 있습니다.  
* **리프 노드(Leaf Node)**: 각 런의 현재 첫 번째 레코드들이 리프 노드에 위치합니다.  
* **루트 노드(Root Node)**: 트리 전체에서 가장 작은(승자) 노드가 위치합니다.  
* **표현 방식**: 이진 트리에 대한 **순차 할당 기법(배열)** 을 사용하여 효율적으로 구현합니다.  

### 12-2. 핵심 작동 원리: 토너먼트 방식  
두 개의 자식 노드 중 더 작은 노드가 승자가 되어 부모 노드로 올라가는 구조입니다.  

1.  **초기 트리 구성**: $k$개의 런에서 각각 첫 번째 데이터를 가져와 리프 노드를 채웁니다.  
2.  **승자 결정**: 형제 노드끼리 비교하여 더 작은 값(승자)을 부모 노드에 기록합니다. 이 과정을 루트까지 반복합니다.  
3.  **결과 출력**: 루트 노드에 도달한 최종 승자가 가장 작은 값이므로 이를 출력합니다.

### 12-3. 한 레코드 출력 후의 재구성 (Reconstruction)  
최종 승자가 출력되어 루트가 비게 되면, 승자가 속했던 런에서 다음 데이터를 가져와 트리를 갱신합니다.  

* **갱신 경로**: 새로 삽입된 리프 노드에서 **루트 노드까지의 경로**를 따라가며 토너먼트를 재수행합니다.  
* **특징**: 전체 트리를 다시 만들 필요 없이, 값이 바뀐 경로상의 노드들만 다시 비교하면 되므로 매우 효율적입니다.  


### 12-4. 복잡도 분석    
* **초기 트리 구성 시간**: $O(k)$ ($k$는 런의 개수)  
* **재구성 시간**: $O(\log_2 k)$  
    * 최종 승자가 나간 후, 새 노드가 루트까지 올라가며 비교하는 횟수는 트리의 높이($\log_2 k$)에 비례합니다.  
* **전체 정렬 시간**: $n$개의 레코드를 정렬할 경우 $O(n \log_2 k)$가 소요됩니다.  

### 12-5. 승자 트리의 문제점 및 한계  
승자 트리는 효율적이지만, 재구성 과정에서 다음과 같은 특징이자 단점이 존재합니다.  

1.  **형제 노드 참조 필수**: 재구성 시 경로를 따라 올라가면서 **항상 형제 노드와 비교**를 수행해야 합니다.  
2.  **패자 데이터의 위치**: 여기서 형제 노드들은 이전 토너먼트에서 해당 단계의 **패자(Loser)**  들입니다.  
3.  **비효율적 접근**: 부모로 올라갈 때마다 형제 노드의 값을 확인하기 위해 메모리 접근이 추가로 발생합니다.  

> **※ 발전 방향**: 승자 트리의 이러한 번거로움(형제 노드 참조)을 해결하기 위해, 부모 노드에 승자 대신 패자를 기록하는 **패자 트리(Loser Tree)**  가 대안으로 사용됩니다.

## 13. 패자 트리 (Loser Tree) 
### 13-1. 핵심 개념
패자 트리는 승자 트리와 마찬가지로 여러 개의 정렬된 런(Run)을 병합하는 데 사용되지만, 토너먼트의 **패자(Loser)** 를 내부 노드에 저장하는 완전 이진 트리입니다.  

* **내부 노드(Non-terminal Node)**: 해당 단계의 토너먼트에서 진(더 큰 값을 가진) 레코드에 대한 포인터를 유지합니다.  
* **노드 0 (Overall Winner)**: 전체 토너먼트의 최종 승자를 표현하기 위해 루트 위에 별도의 **노드 0**을 추가합니다.  
* **비교 방식**: 새로운 데이터가 들어오면 **부모 노드에 저장된 값과 직접 비교**하며 올라갑니다.  

### 13-2. 작동 원리 (Tournament Logic)
1.  **리프 노드 비교**: 두 형제 노드가 경기를 하여 승자와 패자를 가립니다.  
2.  **패자 기록**: 패한 데이터는 현재의 **부모 노드**에 남습니다.  
3.  **승자 이동**: 승리한 데이터는 더 상위 부모 노드로 올라가서, 그곳에 이미 기다리고 있는 **이전 단계의 패자**와 다시 경기를 치릅니다.  
4.  **최종 승자**: 모든 단계에서 이기고 올라온 최후의 승자는 **노드 0**에 저장됩니다.

### 13-3. 구조적 특징 (승자 트리와의 차이)  
| 특징 | 승자 트리 (Winner Tree) | 패자 트리 (Loser Tree) |  
| :--- | :--- | :--- |
| **내부 노드 저장값** | 해당 구간의 승자 | 해당 구간의 **패자** |
| **비교 대상** | **형제 노드**와 비교 | **부모 노드**와 비교 |
| **노드 0 존재 여부** | 없음 (루트가 승자) | 있음 (최종 승자 보관용) |  

### 13-4. 패자 트리의 강력한 장점  
승자 트리에서 지적되었던 문제점들을 해결하여 실질적인 연산 속도를 높였습니다.  

* **형제 노드 접근 불필요**: 재구성 시 부모 노드에 이미 패자가 기록되어 있으므로, 형제 노드가 무엇인지 찾아서 메모리에서 읽어올 필요가 없습니다.  
* **로컬성(Locality) 향상**: 단순히 부모 노드만 참조하며 루트까지 올라가면 되므로 캐시 효율이 높고 제어 흐름이 단순합니다.  


### 13-5. 복잡도 분석
* **초기 트리 구성**: $O(k)$
* **재구성 시간**: $O(\log_2 k)$
    * 승자 트리와 시간 복잡도는 동일하지만, 실제 하드웨어 수준에서의 비교 연산 및 메모리 접근 횟수가 적어 성능상 유리합니다.
* **전체 합병 시간**: $O(n \log_2 k)$ ($n$: 전체 레코드 수, $k$: 런의 개수)
