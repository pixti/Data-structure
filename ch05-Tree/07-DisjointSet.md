## 16. 상호배타적 집합 (Disjoint Set)
### 16-1. 상호배타적 집합의 정의 및 연산

**상호배타적 집합(Disjoint Set)** 이란 이름 그대로 "서로 겹치는 원소가 하나도 없는" 집합들을 의미합니다.   
수학적으로는 두 집합 $A, B$에 대해 $A \cap B = \emptyset$인 상태를 말합니다.  

* **핵심 연산**:  
    * **Union (합집합) 연산**: 두 개의 독립된 집합을 하나의 집합으로 합치는 연산입니다.  
    * **Find (찾기) 연산**: 특정 원소가 주어졌을 때, 이 원소가 어느 집합에 속해 있는지(즉, 집합의 대표자가 누구인지) 찾아내는 연산입니다.  
* **주요 활용**:   
    * 그래프 알고리즘 중 **Kruskal의 최소 신장 트리(MST) 알고리즘**에서 사이클 형성 여부를 확인하기 위해 필수적으로 사용됩니다.  
    * 네트워크 연결성 확인, 이미지 분할 등 다양한 영역에서 활용됩니다.  

### 16-2. 상호배타적 집합을 위한 트리 표현  
상호배타적 집합은 트리 구조를 사용하여 가장 효율적으로 표현할 수 있습니다.   
각 집합은 하나의 트리로 간주하며, **트리의 루트 노드가 곧 그 집합의 대표자(Representative)** 가 됩니다.   

* **구조적 특징**:  
    * 일반적인 트리와 반대로, 자식이 부모를 가리키는 포인터를 가집니다. (위로 거슬러 올라가기 위함)  
    * **루트 노드**: 집합을 대표하는 노드입니다.  
    * **Find 연산의 원리**: 특정 노드에서 시작해 부모를 따라 계속 올라가다 보면 결국 루트 노드에 도달하게 되며, 이 루트가 해당 원소의 소속을 결정합니다.  

### 16-3. 1차원 배열을 이용한 메모리 저장  
트리 구조임에도 불구하고 별도의 연결 리스트나 복잡한 객체 없이 **1차원 배열**만으로 매우 간단하게 구현할 수 있습니다.   
이를 **부모 배열(Parent Array)** 표현법이라고 합니다.  

* **배열 저장 규칙**:  
    * 배열의 **인덱스(Index)**: 원소의 번호 (0, 1, 2, ..., N-1)  
    * 배열의 **값(Value)**: 해당 노드의 **부모 노드 번호**  
    * **루트 노드 판별**:   
        * 루트 노드는 부모가 자기 자신입니다. 즉, `a[i] == i` 인 경우 인덱스 `i` 가 루트입니다.  
        * (루트의 값을 -1이나 집합의 크기를 나타내는 음수값으로 저장하기도 하고, 자기 자신을 가리키는 방식으로 구현하기도 합니다.)

    * 0, 5, 9는 4의 자식이므로 `a[0]=4, a[5]=4, a[9]=4`

## 17. 상호배타적 집합을 위한 트리 연산

### 17-1. Find 연산: 대표 노드(Root) 찾기
Find 연산은 특정 원소 $x$가 속한 집합의 **대표 노드(루트)** 를 찾는 과정입니다.  
두 원소가 같은 집합에 있는지 확인하려면 각각 Find를 수행하여 결과(루트)가 같은지 비교하면 됩니다.

<img width="769" height="447" alt="image" src="https://github.com/user-attachments/assets/4e5a6173-c0fe-4ee2-ad97-501cb4f276ca" />

* **연산 과정 (예시: `find(6)`)**:  
    1. `a[6]`을 확인하여 부모 노드 **2**를 찾음.  
    2. `a[2]`를 확인하여 부모 노드 **7**을 찾음.  
    3. `a[7]`이 **7**이므로(자기 자신), **7이 루트임을 확정하고 반환.**  
* **판단 결과**:  
    * `find(6) = 7`, `find(3) = 7` $\rightarrow$ 두 결과가 같으므로 **6과 3은 같은 집합**입니다.  
    * `find(6) = 7`, `find(9) = 4` $\rightarrow$ 두 결과가 다르므로 **6과 9는 서로 다른 집합**입니다.  

### 17-2. Union 연산: 집합의 통합  
수학적으로 Union 연산은 두 개의 상호배타적 집합 $A$와 $B$ 를 하나의 합집합 $A \cup B$ 로 만드는 연산입니다. 
트리 구조에서는 **한쪽 트리의 루트를 다른 쪽 트리의 자식으로 연결**하는 방식으로 구현합니다.  

<img width="691" height="272" alt="image" src="https://github.com/user-attachments/assets/950ffd31-bad1-4a33-a3d7-33aeeb7d8623" />


* **핵심 아이디어: Union-by-Rank (랭크 기반 합치기)**:  
    * **Rank**: 여기서 랭크는 해당 트리의 **높이(Height)** 를 의미합니다. (노드 수 기반으로 하기도 하며 성능은 비슷함)  
    * **원칙**: 항상 **랭크가 낮은 트리를 랭크가 높은 트리 밑에** 붙입니다. 랭크가 높은 루트가 합쳐진 트리의 '승자(루트)'가 됩니다.    
    * **목적**: 트리가 옆으로 퍼지게 만들어 높이가 커지는 것을 방지함으로써, 이후 Find 연산의 속도를 최적화하기 위함입니다.  

### 17-3. Union-by-Rank의 세부 규칙
1. **두 트리의 랭크가 다를 때**: 랭크가 높은 쪽이 루트가 되며, 합쳐진 후에도 전체 트리의 **랭크(높이)는 변하지 않습니다.**
2. **두 트리의 랭크가 같을 때**: 둘 중 아무나 루트가 되어도 상관없으나, 합쳐진 후 전체 트리의 **랭크가 1 증가**하게 됩니다.
    * 비효율적인 Union은 트리의 높이를 불필요하게 키워 Find 연산 시 루트까지 올라가는 시간을 길어지게 만듭니다.

#### 왜 랭크가 높은 노드가 승자가 되어야 할까?
만약 랭크가 더 낮은 노드(높이가 낮은 트리)가 루트가 되어버리면, 그 아래에 더 높은 트리가 붙으면서 전체 트리의 높이가 무조건 1 증가하게 됩니다.   
하지만 랭크가 높은 노드 아래에 낮은 트리를 붙이면, 마치 큰 가방 안에 작은 가방을 넣는 것처럼 전체 높이를 유지할 가능성이 높습니다.    

#### 왜 Union-by-Rank에서 Rank가 같을 때만 1을 더할까?  
랭크가 다른 두 트리를 합칠 때는 작은 쪽이 큰 쪽 아래로 들어갑니다. 이때 큰 쪽 트리의 전체 높이는 변하지 않습니다.   
하지만 **랭크가 같은(높이가 같은)** 두 트리를 합치면, 누가 위로 가든 전체 트리의 높이는 기존보다 층수가 하나 더 생길 수밖에 없기 때문에 이때만 Rank를 1 높여주는 것입니다.    

Find 연산의 속도는 **루트까지의 거리(높이)** 에 정비례하기 때문에, 최대한 트리를 "납작하게" 유지하는 것이 이 자료구조의 핵심 성능 포인트입니다.   

### 17-4. Union 수행 예제  

<img width="1646" height="543" alt="image" src="https://github.com/user-attachments/assets/8ff7b2ec-2ac4-465e-b739-67abab037fe9" />


* **수행 전**:   
    * 집합 1 (루트 **7**, 랭크 **2**)  
    * 집합 2 (루트 **4**, 랭크 **1**)  
* **연산 수행: `union(7, 4)`**:  
    * 7의 랭크(2) > 4의 랭크(1) 이므로, **7이 승자**가 됩니다.  
    * 4번 노드의 부모를 7로 변경합니다. (`a[4] = 7`)  
* **수행 결과**:  
    * 4가 7의 자식이 되었지만, 7의 기존 높이(2) 안에 4의 트리(높이 1)가 충분히 포함되므로 **전체 랭크는 여전히 2**입니다.  
    * 배열 상태: `a[4]`의 값이 기존 4에서 **7**로 갱신됩니다.
 
## 18. Find 연산의 최적화: 경로 압축 (Path Compression)  
### 18-1. 경로 압축의 정의  
Find 연산을 수행할 때, 방문하는 모든 노드들이 **직접 루트 노드를 가리키도록** 부모 포인터를 갱신하는 기법입니다.  

* **핵심 아이디어**: "한 번 루트를 찾으러 올라간 김에, 거쳐온 모든 노드들을 루트 바로 아래에 붙여버리자!"  
* **효과**: 트리의 높이가 획기적으로 낮아져(Flattening), 이후 동일한 노드나 그 자손들에 대한 Find 연산 속도가 비약적으로 빨라집니다.  

### 18-2. 작동 원리 및 예시

<img width="1598" height="537" alt="image" src="https://github.com/user-attachments/assets/93abfda4-61ed-45f5-944b-45126777d5f5" />

* **연산 전 (`find(1)`)**: 1번 노드에서 루트 10까지 가기 위해 `1 -> 2 -> 7 -> 10`의 경로를 거쳐야 합니다. (경로상의 노드: 1, 2, 7)  
* **연산 수행**: 루트가 10임을 확인한 후, 경로에 있던 노드 1, 2, 7의 부모를 모두 10으로 직접 연결합니다.  
* **연산 후**:  
    * 배열 상태: `a[1]=10, a[2]=10, a[7]=10`으로 갱신됩니다.  
    * **미래의 이득**: 이제 `find(1)`이나 `find(2)`를 호출하면 단 한 번만에 루트 10을 찾을 수 있습니다.  

### 18-3. 경로 압축의 특징 및 주의사항  
1. **지연된 최적화**: 당장 수행 중인 `find(1)` 자체의 시간을 줄여주지는 않지만, **추후 수행될 연산들**의 시간을 단축합니다.  
2. **Rank와 실제 높이의 불일치**:   
    * 경로 압축을 하면 트리의 실제 높이는 낮아지지만, 효율성을 위해 **루트의 Rank(랭크) 값은 수정하지 않습니다.**  
    * 따라서 Rank는 '실제 높이'가 아니라 **'트리가 가질 수 있는 최대 높이의 상한선'** 이라는 의미로 이해해야 합니다.  
    * Rank를 일일이 수정하는 것보다, 그대로 두고 Union-by-Rank를 수행하는 것이 전체적인 상각 분석(Amortized Analysis) 측면에서 더 유리하기 때문입니다.  

## 19. 상호배타적 집합 연산의 수행 시간  
### 19-1. Union 연산의 수행 시간: $O(1)$  
* **전제 조건**: 두 집합의 루트 노드를 이미 알고 있다는 가정하에 수행됩니다.  
* **이유**: 단순히 한 루트의 부모 포인터를 다른 루트로 바꾸고, 필요 시 Rank 값을 1 증가시키는 작업만 수행하므로 상수 시간 $O(1)$이 소요됩니다.  
* **실제**: 보통 `union(x, y)`는 내부적으로 `find(x)`와 `find(y)`를 먼저 호출하므로, 전체 시간은 Find 연산 시간에 종속됩니다.    

### 19-2. Find 연산의 수행 시간: $O(H)$ → $O(\alpha(N))$  
* **기본 복잡도**: 트리의 높이($H$)에 비례합니다. 최악의 경우(편향 트리) $O(N)$이 될 수 있습니다.   
* **최적화 적용 시**: Union-by-Rank와 경로 압축을 모두 사용하면, Find 연산의 수행 시간은 아커만 함수의 역함수 $\alpha(N)$ 에 비례하게 됩니다.  
    * $\alpha(N)$ 은 $N$ 이 우주의 원자 수보다 많아도 **4 이하**의 값을 가질 정도로 매우 느리게 증가하는 함수입니다.  
    * 따라서 실질적으로 **실행 시간은 거의 상수 시간 $O(1)$**이라고 보아도 무방합니다.  

### 19-3. 종합 분석 요약  

| 연산 종류 | 기본 구현 (배열/트리) | 최적화 구현 (Rank + 경로 압축) |  
| :--- | :--- | :--- | 
| **Union** | $O(N)$ (배열 시) / $O(H)$ (트리 시) | 사실상 $O(1)$ | 
| **Find** | $O(1)$ (배열 시) / $O(H)$ (트리 시) | 사실상 $O(1)$ |
