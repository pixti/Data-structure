# 1. 연결 리스트 (Linked List)

LinkedList는 사용자 정의 클래스인 **Node**를 사용하여 **List ADT**를 구현한 자료구조입니다.   
ArrayList의 치명적 단점이었던 '중간 삽입/삭제 시 데이터 시프팅( $O(N)$ )' 문제를 해결하기 위해 고안되었습니다.

* **특징**: 크기를 동적으로 조절할 수 있으며, 데이터의 삽입과 삭제가 빈번하게 일어나는 환경에서 압도적인 효율을 자랑합니다.

---

## 1-1. 노드 (Node) : 연결 리스트의 원자(Atom)
노드는 리스트를 구성하는 가장 작은 단위이자 **사용자 정의 클래스** 객체입니다.

1. **데이터 필드 (item)**: 실제 저장하려는 값 (Integer, String, 객체 등).
2. **링크 필드 (next)**: 다음 노드가 메모리 어디에 있는지 가리키는 **레퍼런스(주소값)**.

> **자기 참조(Self-referential) 구조**: 노드 내부의 `next`가 자신과 같은 타입인 `Node` 객체를 가리키는 형태입니다. 이를 통해 `nodeA.next = nodeB`와 같이 기차 칸을 잇듯 데이터를 연결합니다.

## 1-2. 연결 리스트의 탐색: "연속성의 붕괴"
배열은 물리적 주소가 연속적이라 인덱스만 알면 $O(1)$로 주소를 계산했지만, 연결 리스트는 다릅니다.

* **물리적 비연속성**: 노드들은 메모리(Heap) 곳곳에 흩어져 존재합니다. (예: A는 100번지, B는 500번지, C는 250번지)
* **포인터 추적 (Pointer Following)**: 컴퓨터는 A의 `next`를 열어봐야만 B의 위치를 알 수 있고, B를 열어봐야 C를 알 수 있습니다.
* **시간 복잡도**: 특정 데이터를 찾으려면 처음부터 끝까지 실을 따라가야 하므로 $O(N)$이 소요됩니다.

---

## 1-3. 단일 연결 리스트 (Singly Linked List)
가장 기본적인 형태로, 각 노드가 **다음 노드의 주소(Next)** 하나만을 가집니다.

* **유연한 확장**: 주소만 계속 이으면 무한하게 확장 가능합니다.
* **연결의 끝 (Terminator)**: 리스트의 마지막 노드는 더 이상 가리킬 대상이 없으므로 `next` 필드에 `null`을 저장하여 끝을 알립니다.

## 1-4. 헤드 (Head) : 리스트의 시작점
연결 리스트는 '앞'을 보는 눈이 없습니다. 오직 내 다음이 누구인지만 알 뿐입니다. 
따라서 전체 리스트를 관리하려면 **최초의 노드를 가리키는 `Head` 변수**가 반드시 필요합니다.

* `Head == null`: 리스트가 텅 비어있음을 의미합니다.

## 1-5. 삽입(Insertion) 로직: "안전하게 뒷부분부터 잡기"
연결 리스트는 한 줄의 실로 연결되어 있습니다. 끊는 순서가 틀리면 데이터가 메모리 미아가 되어 **Garbage Collector**에 의해 삭제될 수 있습니다.

**[A와 B 사이에 New를 삽입하는 과정]**
1. **Step 1 (뒷부분 연결)**: `New.next = A.next;` 
   - 새 노드가 먼저 B를 잡게 합니다. 이제 A와 New 둘 다 B를 잡고 있어 안전합니다.
2. **Step 2 (앞부분 수정)**: `A.next = New;`
   - A의 손을 놓아 새 노드를 잡게 합니다. 연결 완료.

* **맨 앞/뒤 삽입**: 원리는 동일합니다. 맨 앞은 `Head`를 갱신하고, 맨 뒤는 기존 끝 노드의 `next`를 `null`에서 새 노드로 바꿔줍니다.

## 1-6. 삭제(Deletion) 로직: "소외(Isolate) 시키기"
삭제하려는 노드를 리스트의 흐름에서 제외하면 됩니다.

* **방법**: 삭제할 노드의 이전 노드(prev)가 가리키는 `next`를 삭제할 노드의 `next.next`로 변경합니다.
* **결과**: 연결에서 소외된 노드는 참조하는 곳이 없어지며, Java의 **Garbage Collector**가 알아서 수거해 갑니다.


## 1-7. 시간 복잡도 요약

| 연산 유형 | 시간 복잡도 | 설명 |
| :--- | :--- | :--- |
| **접근/탐색** | $O(N)$ | 인덱스가 없으므로 처음부터 순차 탐색 필수 |
| **삽입/삭제 (위치 알 때)** | **$O(1)$** | `prevNode`를 알면 주소값만 바꾸면 끝 |
| **삽입/삭제 (위치 모를 때)** | $O(N)$ | 위치를 찾는 탐색(O(N)) + 연산(O(1)) |

* **최고 효율**: **맨 앞** 삽입/삭제 ( $O(1)$ ) - `Head`만 건드리면 됨.
* **최저 효율**: **맨 뒤** 삽입/삭제 ( $O(N)$ ) - 맨 뒤가 어디인지 끝까지 가봐야 함.


## 1-8. 더미 헤드 노드 (Dummy Head Node)
빈 리스트일 때나 맨 첫 번째 노드를 건드릴 때 발생하는 예외 처리를 없애기 위한 기법입니다.

* **구조**: 아무 데이터도 담지 않는 '가짜 노드'를 항상 `Head` 바로 뒤에 둡니다.
* **장점**: 모든 실제 데이터 노드들이 '앞 노드'를 가지게 되어, 삽입/삭제 로직에 **일관성**이 생깁니다.
* **판단**: `Head.next == null`이면 빈 리스트로 간주합니다.

## 1-9. 테일 (Tail) : 마지막 칸 전용 내비게이션
맨 뒤에 데이터를 넣을 때마다 끝까지 달려가는( $O(N)$ ) 비효율을 해결하기 위한 아이디어입니다.

* **장점**: 마지막 노드를 가리키는 `Tail` 변수를 두면, 탐색 없이 즉시 맨 뒤 삽입이 가능해져 **O(1)** 로 단축됩니다.
  
* **단점 1 (관리 부담)**: 삽입/삭제 때마다 `Tail`이 바뀌었는지 체크해야 해서 코드가 복잡해집니다.
* **단점 2 (삭제의 한계)**: 맨 뒤 노드를 **삭제**할 때는 여전히 **O(N)** 입니다. 마지막 노드를 지운 후 `Tail`을 한 칸 앞으로 옮겨야 하는데, 단방향 리스트는 '내 앞'이 누구인지 모르기 때문에 다시 처음(Head)부터 훑어 내려와야 하기 때문입니다.
